# 2026-04-23 — Arreglo integral del pipeline nativo de descargas (Android)

## Contexto

Tres síntomas en producción:

1. Descargas no se ejecutaban realmente (Media3 bajaba solo el `.m3u8` playlist y "completaba").
2. UI de descargas no se actualizaba con el progreso ni con transiciones de estado.
3. Notificaciones sí funcionaban (pintadas por `KumoriyaDownloadService` directo desde `DownloadManager`).

Auditoría rápida confirmó causa-raíz en 3 capas: la `DownloadListener` nativa no emitía
`state` — solo `progress` —, el servicio Dart tenía un guard `status == downloading`
que nunca se cumplía, y `DownloadRequest.Builder` se usaba sin `MimeType` para HLS
(media3 trataba todo como progressive). Además el `eventSink` se recreaba cada vez
que Flutter resuscribía el EventChannel, con riesgo de perder eventos tempranos.

## Cambios

### Kotlin (plugin `kumoriya_exoplayer`)

- `DownloadListener.kt` — sink ahora es `@Volatile var` con método `setSink(sink)`.
  Eventos que ocurren antes de que Flutter suscriba se guardan en un buffer circular
  (máx 64) y se drenan al adjuntar el sink. Además se emite evento `state` separado
  cada vez que cambia el Media3 `Download.state`, con dedupe por taskId.
- `KumoriyaExoPlayerPlugin.kt` — deja de reemplazar la instancia del listener en cada
  `onListen`; delega a `setSink`. Detecta si la URL ya trae `_kumoriya_headers` y
  no lo duplica. Rutea `isHls=true` al nuevo `HlsDownloadPreparer`. Llama a
  `resumeDownloads()` tras crear el DM singleton para limpiar estado persistido.
- `HlsDownloadPreparer.kt` (nuevo) — corre en el `ioExecutor` del plugin,
  `DownloadHelper.forMediaItem(...)` con `MimeTypes.APPLICATION_M3U8`, enumera y
  enqueua segmentos reales. Errores se surface via callback para que el listener
  los emita al sink como evento `failed`.
- `DownloadCleanupUtil.kt` (nuevo) — helpers `cleanupSimpleCache` y
  `cleanupFragmentedSegments` extraídos para reusar en listener y servicio.
- `KumoriyaDownloadService.kt` — `ACTION_CANCEL` ahora limpia cache y segmentos
  temporales del taskId.
- `DownloadManagerProvider.kt` — expone `createHttpDataSourceFactory()` para que
  el HlsDownloadPreparer fetch el manifest conservando los headers custom.

### Dart (`apps/kumoriya_app`)

- `native_download_manager_service.dart` — handler añade rama `case 'state'` que
  llama a `_applyStateTransition` (única fuente de verdad para
  `pending → downloading → completed/failed`, persiste en Drift y emite
  `DownloadStatusChange` con `oldStatus` + `newStatus`). Se quitó el guard
  `status == downloading` del `_updateProgressInDrift` — ahora acepta cualquier
  status no terminal. `cancelled` y `cancelDownload()` ahora borran el task de
  Drift con semántica delete (aprobada por el usuario).
- Tests: reemplaza el archivo stale `native_download_manager_service_test.dart`
  con 4 casos reales — transición pending→downloading, persistencia de bytes,
  cancel = delete, no transición cuando status ya es downloading. 4/4 pasan.

Causa: el plugin compilaba contra Media3 **1.4.1**, pero el APK final resuelve **1.9.2** via transitive de `media_kit_libs_android_video`. ## 16:30 — Cola de descargas con cap de concurrencia

**Category:** feature
**Affected files:**
- `packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/downloads/DownloadEngine.kt`

**Summary:**
El motor nativo corría TODAS las descargas encoladas en paralelo (una coroutine por task, sin gate). Con 10+ tareas saturaba ancho de banda y creaba 10 conexiones HTTP paralelas. Introducido `MAX_CONCURRENT = 3` + FIFO `waitingQueue` con lock (`queueLock`) que sampla `jobs.size` atómicamente. Nuevo método `drainQueue()` promueve de la cola cuando se libera un slot (llamado en `finally` del job, `cancel`, `pauseAll`, etc.). Tareas waiting emiten `PENDING` al Dart. `pause/cancel` saben desencolar sin cancelar Job inexistente. `autoPauseAll` también vacía la cola hacia `DISCONNECTED` para auto-resume en reconexión. `drainQueue` corre ANTES de `onTaskFinished` para que el debouncer del FGS no vea "sin trabajo" entre el release del slot y la promoción del siguiente.

**Linked phase:** Fase 4 (post-release hardening)

## 17:05 — Auto-resume de descargas en cold start

**Category:** fix
**Affected files:**
- `apps/kumoriya_app/lib/src/features/downloads/application/native_download_backend.dart`

**Summary:**
Bug: con descargas en cola, si el usuario cerraba la app (o el sistema mataba el proceso), al reabrir no había forma de reanudar nada — Drift seguía con status `downloading`/`pending`/`remuxing`/`disconnected` pero el motor nativo estaba vacío. Cualquier pause/resume era no-op y no se emitía ningún progreso. Se llama en `initialize()` y `restoreQueue()`; idempotente en warm start porque el engine skippea tasks activos.

## Sync coordinator event-driven (tarde)

### Contexto

Hasta hoy el push al servidor dependía de un botón "Sync Now" en `ProfilePage` y de
la migración anonymous→auth al loguear. Los enqueues sí eran event-driven (los
`SyncAware*Store` ya escribían en la cola tras cada cambio de favoritos, progreso,
historial y preferencias), pero la cola podía quedarse meses sin drenarse. Peor:
el backend solo garantiza durabilidad tras un segundo push que recibe
`durable_until` (ver `HttpSyncService`), así que un usuario que nunca pulsaba el
botón y reinstalaba la app perdía datos.

### Diseño

Se evaluó código nativo Android (BroadcastReceiver + JobService) y se descartó:
Workmanager ya está registrado para el worker de episodios, así que reutilizamos
esa infraestructura. Nuevo `SyncCoordinator` en Dart con 6 triggers:

1. **Escritura local debounced (500 ms)** — callback opcional `onEnqueued` en
   `SyncAwareLibraryStore` / `SyncAwareProgressStore`; `storage_providers.dart` lo
   cablea a `ref.read(syncCoordinatorProvider).notifyLocalWrite()`. Cero cambios
   en el contrato `kumoriya_sync` (la alternativa de un `Stream<void>
   watchEnqueues()` queda como TODO para cuando aparezca un segundo consumidor).
2. **AppLifecycleState.resumed** — `AppLifecycleListener` en `KumoriyaApp`; si
   `lastSyncAt` > 6 h hace `fullSync`, si no solo `pushPending`.
3. **Connectivity regained** — `connectivity_plus` con listener que solo dispara
   en transición offline→online (evita spam por cambios wifi↔celular).
4. **AppLifecycleState.paused** — programa `Workmanager.registerOneOffTask` con
   constraint `NetworkType.connected` y `existingWorkPolicy: replace` si hay
   entradas pendientes.
5. **Pre-logout flush** — `AuthStateNotifier.logout()` llama
   `coordinator.flushBeforeLogout()` bloqueante con timeout 5 s antes de wipe;
   cualquier fallo se swallow y el logout sigue.
6. **Periodic Workmanager 12 h** — fallback absoluto en `main.dart` para el caso
   de app cerrada durante días con cola pendiente. Task `kumoriya.push_pending_sync`
   comparte dispatcher con `check_new_episodes` (switch por nombre).

Reglas transversales en el coordinador:

- Lock de push único en vuelo con flag `_dirty` que reprograma al terminar.
- Offline skip: si `connectivity.isConnected == false`, no se llama a `pushPending`
  (evita envenenar `retryCount` en la queue — cada fallo hoy cuenta para el
  límite de 8 intentos antes de marcar `poisoned`).
- Backoff exponencial 30 s → 2 m → 5 m → 15 m tras fallos; `notifyAppResumed`,
  `notifyConnectivityRegained`, `triggerManual` y `flushBeforeLogout` lo bypassan.
- Auth guard en todos los triggers (no-op si `!isAuthenticated()`).

### Worker de push en Workmanager

`push_pending_sync_worker.dart` corre en isolate aparte: construye su propio
`SecureTokenStore`, `AuthenticatedHttpClient`, `AppDatabase` y `HttpSyncService`
para llamar `pushPending` y cerrar la DB en `finally`. No hace pull ni toca
caches; el `SyncCoordinator` del isolate principal se encarga del `fullSync`.

### Cambios

**Código nuevo:**
- `apps/kumoriya_app/lib/src/shared/sync/sync_coordinator.dart`
- `apps/kumoriya_app/lib/src/workers/push_pending_sync_worker.dart`
- `apps/kumoriya_app/test/shared/sync/sync_coordinator_test.dart` (9 casos)

**Modificado:**
- `apps/kumoriya_app/lib/src/shared/sync/sync_aware_library_store.dart` — param
  opcional `onEnqueued` + llamadas en cada `enqueue`.
- `apps/kumoriya_app/lib/src/shared/sync/sync_aware_progress_store.dart` — idem.
- `apps/kumoriya_app/lib/src/shared/sync/sync_providers.dart` — nuevo
  `syncCoordinatorProvider` con `schedulePushJob` solo en Android.
- `apps/kumoriya_app/lib/src/shared/storage_providers.dart` — cablea callback en
  ambos stores.
- `apps/kumoriya_app/lib/src/shared/auth/auth_providers.dart` — pre-logout flush.
- `apps/kumoriya_app/lib/src/app/kumoriya_app.dart` — ahora `ConsumerStatefulWidget`,

- `apps/kumoriya_app/lib/src/workers/check_new_episodes_worker.dart` — dispatcher

- `apps/kumoriya_app/pubspec.yaml` — `connectivity_plus: ^6.1.0`.

## Fase A: pull independiente en el SyncCoordinator

### Contexto

Tras entregar el coordinador event-driven, el pull quedó expuesto como un
flanco débil: la única manera de traer cambios de otros dispositivos era vía
`fullSync` (login, botón manual, o resume con `lastSyncAt > 6 h`). Un usuario
con dos dispositivos que los abría varias veces al día no veía cambios del otro
hasta el día siguiente.

Se evaluaron tres alternativas para resolverlo de raíz:

- **FCM silent push desde el backend** (Fase B futura): mejor tradeoff
  batería/latencia en Android, pero requiere tocar `kumoriya-api` (device
  registry + fanout) y solo cubre Android/iOS.
- **SSE / WebSocket persistente**: descartado. Android mata sockets en Doze, y
  mantenerlo vivo en foreground es un coste alto para eventos tan poco
  frecuentes como un toggle de favorito.
- **Solo cliente (esta slice)**: `triggerPull` independiente + periódico en
  foreground + pull en cada resume. Cero cambios en API. Cierra el 80% del hueco
  sin tocar backend.

Elegida la tercera para aterrizar valor inmediato y dejar `triggerPull` listo
como hook para la Fase B FCM.

### Verificación de self-echo

Revisado `@/home/reny/Projects/Kumoriya/kumoriya-api/internal/service/sync_service.go:133`
y `repository/sync_repo.go`. El server filtra por `updated_at > since`. Después
de cada push+pull, el cliente persiste `_lastSyncAt = pullResponse.serverTime`
(`HttpSyncService` línea 231), y `serverTime` siempre es ≥ el `updated_at` más
alto que el cliente acaba de subir. Por tanto, en el siguiente `pullSince`, el
servidor ya no devuelve las filas recién pusheadas por este mismo dispositivo.
**No hace falta device_id para Fase A.** Para Fase B sí habrá que añadirlo para
evitar que un dispositivo se notifique a sí mismo vía FCM.

### Cambios

**Coordinador (`sync_coordinator.dart`):**

- `triggerPull()` público: delega a un nuevo `_runPull` con lock `_pulling`
  separado, offline-skip y respeto al backoff (override con
  `bypassBackoff: true` para triggers explícitos).
- `notifyAppResumed()` reescrito:
  - Si `lastSyncAt` > 6h → `fullSync` como antes.
  - Si no, `pullSince` (salvo que el último pull sea < 30 s) + push si queue
    tiene pendientes.
  - Arranca timer periódico de 30 min mientras app esté en foreground.
- `_foregroundPullTimer` cancelado en `dispose()` y reprogramado en cada
  resume.

**Bug corregido de paso:** el lock `_pushing`/`_pulling` se seteaba *después*
del `await _connectivity.isConnected`, dejando una ventana donde dos llamadas
concurrentes podían pasar el guard. Ahora el flag se claim síncrono antes de
cualquier await, igual para push y pull.

### Validación

```bash
flutter test test/shared/sync test/anonymous_to_auth_migration_test.dart
# 13/13 passed
```

Nuevos tests: resume con queue vacía solo pulea; resume rápido no duplica pull
(debounce 30 s); `triggerPull` no-op sin auth; lock pull colapsa llamadas
solapadas.

### Riesgos residuales

- Timer periódico de 30 min en foreground cuesta ~1 request/30 min por usuario
  activo. Si el user abre 5 h seguidas = 10 pulls; aceptable dado que el server
  tiene cache de snapshots (`sync_cache.go` con TTL).
- Sin FCM todavía: si el user A edita y el user A también abre el dispositivo
  B en los siguientes 29 min sin resume, no verá el cambio. Este hueco lo cubre
  Fase B.

### Contexto

Hasta hoy el pipeline de release generaba un solo APK universal (fat) de ~90MB que
contenía las 4 variantes de ABI (arm64-v8a, armeabi-v7a, x86_64, x86). Un usuario
con un teléfono arm64 descargaba nativos de 3 ABIs que nunca iba a ejecutar.

### Decisiones

- **ABIs a publicar**: `arm64-v8a`, `armeabi-v7a`, `x86_64`, `universal` (fallback).
- **Updater in-app**: auto-detecta `Build.SUPPORTED_ABIS[0]` vía `device_info_plus`
  y elige el split correspondiente. Fallback a `universal` si la detección falla.
- **Retrocompatibilidad del JSON**: `android.url` sigue apuntando al universal para
  que clientes viejos que solo leen `url` sigan funcionando. Los nuevos campos
  son `universal` y `abis` (mapa `abi → {url, file_name, size_bytes, sha256}`).

### Cambios

**Go backend (`kumoriya-api`):**
- Migration `013_app_releases_android_abis.sql` — nueva tabla hija
  `app_release_android_artifacts(tag, abi, url, file_name, r2_key, size_bytes,
  sha256)` con backfill del universal desde `android_url`.
- `internal/model/release.go` — `AndroidArtifacts` con `Universal *ReleaseArtifact`
  y `ABIs map[string]*ReleaseArtifact`. Métodos `Primary()` (universal → mejor
  ABI) e `IsEmpty()`.
- `internal/repository/release_repo.go` — upsert escribe en ambas tablas dentro
  de la misma tx; lectura hace JOIN implícito cargando child rows por tag. Las
  columnas legacy `android_url/file_name/r2_key` se mantienen como mirror del
  universal para rollback limpio.
- `internal/service/release_service.go` — nuevos structs `AndroidLatestManifest`
  (con `Universal` + `ABIs`) y `AndroidDownloadsResponse` (con `UnmarshalJSON`
  que acepta tanto shape flat legacy como nested nuevo). `buildLatestManifest`
  y `buildReleaseDetails` propagan los 4 artefactos. Publish input valida al menos
  un artifact y promociona legacy `url` → `universal`.
- `internal/handler/release_handler_test.go` — dos tests nuevos:
  `TestReleasePublishWithABIsplits` (verifica que `/releases/latest` y
  `/releases/:tag` devuelven `universal` + 3 ABIs) y
  `TestReleasePublishLegacyFlatAndroidShape` (verifica backward-compat con
  publish scripts viejos que envían solo `url/file_name/r2_key`).

**Publish script (`scripts/windows/publish-r2-release.ps1`):**
- Build ahora corre `flutter build apk --release` (universal) seguido de
  `flutter build apk --release --split-per-abi --target-platform android-arm,android-arm64,android-x64`.
- Renombrado: `kumoriya-<version>-<abi>.apk` / `kumoriya-<version>-universal.apk`.
- SHA-256 + `size_bytes` calculados por archivo con `Get-FileHash`.
- Subida: los 4 APKs a `artifacts/android/<tag>/` con content-type
  `application/vnd.android.package-archive`.
- `update.json` y body de `/internal/releases/publish` ahora incluyen
  `android.url` (= universal) + `android.universal` + `android.abis`.

**Flutter updater (`apps/kumoriya_app`):**
- `AppUpdateService.checkForUpdate` — para `platformKey == 'android'`, llama a
  `_resolveAndroidDownloadUrl(platformData)`. Lee `abis` del manifest, itera
  `deviceInfo.supportedAbis` buscando un match en el mapa. Si no hay match o
  el host no es Android, cae a `universal` y luego al `url` legacy.
- `AvailableUpdate` ahora lleva `sizeBytes` opcional (se pasa al UI para mostrar
  tamaño estimado antes de la descarga).
- Parámetros `currentVersion` y `platformOverride` inyectables para testing.
- Test `app_update_service_abi_test.dart` — 2 casos: legacy flat manifest y
  manifest con ABIs (universal fallback en test runner no-Android).

### Validación

- `go test ./internal/handler/ ./internal/service/ ./internal/model/ -count=1` — all green.
- `cd apps/kumoriya_app && flutter test test/src/features/app_update/app_update_service_abi_test.dart` — 2/2 passed.
- `flutter analyze lib/src/features/app_update/application/app_update_service.dart` — 0 errores.

### Riesgos residuales

- SHA-256 no se consume aún en ningún lado (app no verifica). Es infraestructura
  para un verificador futuro (signature check o integridad post-download).
- La página web de descargas queda por adaptar al nuevo shape (4 URLs). El API
  ya entrega los datos correctos.
- Si un publish script antiguo envía `downloads.android.url` sin `universal` ni
  `abis`, el service lo promociona a `universal`. Si un script nuevo envía ambos,
  `universal` tiene precedencia.
