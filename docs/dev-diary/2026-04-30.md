# 2026-04-30 · Dev diary

## S2 (M2) — Configurable base URL + fallback list contract

Closed in a single slice covering S2.A + S2.B + S2.C. The infrastructure that
unblocks every host-rotating source plugin (Olympus, ComicK, Ikigai, future
mirrors-of-the-month) now lives in the workspace.

### What landed

**S2.A — `kumoriya_source_runtime` (new leaf package)**

- `MirrorList` — ordered, non-empty list of equivalent base URIs. Normalizes
  every entry to end with a trailing slash (so callers can `Uri.resolve`
  cleanly). `withPreferred(uri)` promotes a user override to the front and
  dedupes.
- `MirrorRotator.run<T>(closure)` — walks the list once per call, returning
  the first success; on transport-classified failure rotates to the next
  mirror; on non-transport failure rethrows verbatim without burning further
  mirrors (parse/4xx/5xx-with-body never silently degrade).
- `TransportFailure.classify(error)` — narrow allowlist:
  `SocketException`, `HttpException`, `TimeoutException`,
  `http.ClientException`. Status-code interpretation explicitly stays with
  the caller (a 404 on a UUID lookup is not a mirror problem).
- 19/19 tests across MirrorList / MirrorRotator / TransportFailure.

**S2.B — MangaDex adopts the rotator**

- `MangaDexSourcePlugin` constructor now accepts a `MirrorList? mirrors`
  parameter (with `Uri? baseUri` retained for backward-compat with existing
  tests). Default is single-entry `https://api.mangadex.org/`.
- Internal `_getJson` runs through `MirrorRotator.run`, so transport
  failures fall through to the next mirror, and the existing
  `mangadex.transport_failed` envelope is preserved when *every* mirror
  fails.
- 3 new regression tests: rotation on `SocketException`, no-rotation on
  `FormatException`, error envelope when all mirrors are exhausted.

**S2.C — Persistence + Settings UI**

- Drift schema bumped to **v22**. New table `plugin_base_url_override`
  (PK pluginId, baseUrl, updatedAt). One row per plugin — absent row means
  "use manifest default order".
  - Also fixed a latent bug from S1.E: `schemaVersion` was still `20` even
    though a `from < 21` migration block existed. Bumping to `22` lets both
    pending migrations apply for users on schema 20.
- `PluginBaseUrlOverrideStore` contract + `DriftPluginBaseUrlOverrideStore`
  implementation. Validates absolute http(s) URLs at the store layer
  (defense in depth — same check at the widget). 7/7 store tests.
- Riverpod wiring: `pluginBaseUrlOverrideStoreProvider` + reactive
  `pluginBaseUrlOverridesProvider` (Future of `Map<String, Uri>`).
  `mangaSourcePluginsProvider` now reads the override snapshot and feeds
  it into `MangaDexSourcePlugin(mirrors: MirrorList([override, default]))`.
  Loading/error states degrade to manifest defaults — overrides never
  block the catalog.
- New `PluginBaseUrlOverridesPage` — one card per plugin, shows
  `displayName`, `manifest.id`, manifest `baseUrls`, current override
  (when set), text field for the new URL, **Save** / **Reset** buttons,
  inline validation + snackbars on success/failure.
- Entry in Settings → App section ("Plugin base URLs (advanced)" /
  "URLs base de plugins (avanzado)").
- 13 new l10n keys × 2 locales (en + es).

### Residual risk

- **Manifest mirror semantics still loose**. `PluginManifest.baseUrls` for
  MangaDex declares `['https://mangadex.org', 'https://api.mangadex.org']`
  — those are *not* equivalent endpoints (web vs API). The new MirrorList
  used at runtime stays surgical (`[override, api]` only) and never
  consumes manifest baseUrls as mirror sources, but the manifest field
  remains a documentation-only hint until a future cleanup retypes it.
- **Plugin discovery is hardcoded at one call site**. The override page
  iterates `mangaSourcePluginsProvider` only — anime sources won't show up
  here until they adopt the same provider shape (deferred to S3+).
- **No health probe / circuit breaker yet**. A perpetually-down primary
  still costs one round-trip per request before the rotator falls through.
  This is intentional for S2 (matches the plan); M4 (S10) layers HEAD
  probes and recency-weighted reordering on top.

## S3 — `kumoriya_source_olympus`

First LatAm source plugin. Recon via Playwright (popunder ads on the live
mirrors made the live snapshot harder than it should have been). Olympus
turned out to be Nuxt 3 SSR + a mostly-unauthenticated REST API on a
paired `dashboard.*` subdomain — much more tractable than pure HTML
scraping.

### Endpoints discovered

| Endpoint | Where | Use |
|---|---|---|
| `GET {web}/api/series/list` | web frontend | Full catalog (~840 entries, comics + novelas mixed). Search filters in-memory. |
| `GET {dashboard}/api/series/{slug}/chapters?page=N&direction=desc&type=comic` | dashboard | Paginated chapters. Returns `{data:[{id, name, published_at, team:{name}}], links, meta}`. |
| `GET {web}/series/comic-{slug}` | web frontend | HTML page; manga detail in `__NUXT_DATA__`. |
| `GET {web}/capitulo/{chapterId}/comic-{slug}` | web frontend | HTML page; pages array in `__NUXT_DATA__`. |

The dedicated detail endpoint `GET /api/series/{slug}?type=comic` requires
auth (returns 401), but the Nuxt SSR pre-bakes the same payload into the
HTML, so we scrape the script tag instead.

### What landed

**`kumoriya_source_runtime` reuse.** Olympus owns two `MirrorRotator`s —
one for the web frontend pair, one for the dashboard API pair. A
transport failure on either layer falls through to the next mirror pair
without exposing rotation upstream. Default order:

```text
web:        olympusbiblioteca.com → olympusscanlation.com → tomanhua.com
dashboard:  dashboard.<above>     (1:1 derived)
```

**`NuxtDataDecoder` (internal helper).** Nuxt 3 serializes its SSR state
as a flat array where every slot may be either a primitive value or an
integer reference into the same array. The decoder dereferences indices
recursively, distinguishing slot-content (terminal even when an int — a
literal manga id like `1445`) from in-container ints (references that
need a fresh `_deref`). Unwraps `["ShallowReactive", X]` /
`["Reactive", X]` / `["Set"]` reactivity tags transparently. Cycle-safe.
9/9 tests including two real fixtures captured live.

**Plugin contract**:
- `search` — in-memory filter over the catalog (1h TTL cache, single
  primary fetch, paginated client-side). Filters out `type: "novela"`
  rows so the manga catalog stays clean.
- `getLatestUpdates` — empty success per contract. Olympus does not
  expose a clean unauthenticated latest-feed endpoint and the contract
  explicitly allows this degradation.
- `getMangaDetail` — fetches the SSR HTML and walks the decoded
  `__NUXT_DATA__` tree to the inner `data` payload, picking
  `name`, `summary`, `slug`, `cover`, `genres[].name`, `team.name`,
  `status.name`. Status mapping: `Activo` → releasing, `Finalizado` →
  finished, `Pausado por el autor (Hiatus)` → hiatus, `Cancelado` →
  cancelled. Format defaults to `manhwa` (Olympus is Korean-source
  heavy); country `kr`.
- `getChapters` — paginated dashboard JSON. Chapter `name` parsed as
  double so fractional chapters (`74.1`, `66.5`) round-trip. All
  chapters tagged with `language: 'es'` and `scanlator: 'Olympus'` (the
  in-house scan team).
- `getChapterPages` — fetches the reader HTML, walks `__NUXT_DATA__`
  to the chapter's `pages` array (ordered list of WebP URLs on the
  dashboard CDN), wraps as `SourcePage` with sequential indices.

**App-side wiring** (`manga_catalog_providers.dart`):
- `mangaSourcePluginsProvider` now returns a list of two plugins
  (MangaDex + Olympus) in registration order. Composite repo dedups
  ties by registration priority — MangaDex wins on conflicts, which is
  the desired behavior since MangaDex has cleaner cross-tracker IDs.
- `_buildOlympus(overrides)` honors the per-plugin URL override from
  S2.C. Web override is interpreted as the preferred frontend host; the
  matching dashboard host is derived by prefixing `dashboard.`.
- Olympus appears in the source picker UI (S1.E) automatically because
  `availableSourcesProvider` already iterates `mangaSourcePluginsProvider`.

### Residual risk

- **Cloudflare interactive challenge.** Recon notes flagged Olympus as
  "light Cloudflare; passes with realistic UA". The plugin sends a
  Chrome-like UA + `Accept-Language: es,en;q=0.8` and that survived
  every Playwright session, but if Olympus tightens its challenge the
  Settings UI lets a user pin a working mirror or swap the http client
  (e.g. `cronet_http`) at the app layer.
- **Catalog cache TTL is in-memory only.** A cold start always re-fetches
  the 174-KB catalog. If we want offline-friendly behavior or cross-app
  sharing, that wants a future Drift-backed cache (out of scope here).
- **Dashboard host derivation assumes the `dashboard.*` convention.**
  Verified for `olympusbiblioteca.com`; assumed (not lab-verified) for
  `olympusscanlation.com` and `tomanhua.com`. If those mirrors deploy
  the API under a different prefix, the dashboard rotator will rotate
  through them and fail; the user override is the escape hatch in the
  meantime.
- **`getLatestUpdates` is intentionally empty.** Composite home feed
  will skip Olympus entirely until a viable endpoint is found or the
  `/capitulos` SSR HTML scraper lands.
- **Status mapping is text-only.** The Nuxt payload also has a numeric
  `id` (1=Activo, 3=Hiatus, 4=Finalizado, …) which would be more stable
  than localized strings. Today the plugin trusts the localized name
  with reasonable fallbacks; if Olympus rotates labels we'll miss-classify
  silently. Acceptable for now.

## S4 — `kumoriya_source_inmanga`

Second LatAm source plugin, single-mirror, much simpler than Olympus.
Recon via Playwright took ~10 minutes — InManga is an old-school
ASP.NET MVC site with two unauthenticated JSON endpoints and an SSR
HTML fallback.

### Endpoints discovered

| Endpoint | Purpose |
|---|---|
| `GET /manga/GetQuickSearch?name={q}` | Free-text search. Quirky double envelope: `{"data": "<inner-json-string>"}`. Inner JSON is the actual `{message, success, result:[{Name, Identification, ThumbnailPath, Sinopsis, BroadcastStatusDescription, …}]}`. |
| `GET /chapter/getall?mangaIdentification={uuid}` | All chapters in one shot. Same double envelope. |
| `GET /ver/manga/_/{uuid}` | Manga detail HTML — synopsis, status text, cover image. The slug segment is SEO-only; controller routes by UUID, so we always pass `_` as the placeholder slug. |
| `GET /ver/manga/_/{chapterNumber}/{chapterUuid}` | Chapter reader HTML; page UUIDs live inside `<select id="PageList">` options. |

Page CDN URL pattern (built client-side):

```text
https://cdn1.intomanga.com/i/m/{mangaUuid}/c/{chapterUuid}/o/{pageUuid}.jpg
```

### What landed

- **Double-envelope decoder.** Helper `_readInner(envelope)` parses
  `{"data": "<json-string>"}` → `Map<String, Object?>` with non-fatal
  fall-through to `inmanga.search_bad_envelope` /
  `inmanga.chapters_bad_envelope` typed failures.
- **Search** filters by `query` and paginates client-side (server returns
  no pagination params; queries are short so this is fine).
- **getMangaDetail** scrapes the SSR HTML page using anchor-based
  string slicing — no HTML parser dependency added. Title via regex on
  `<h1>...</h1>`, cover via attribute regex on
  `<img ... src="https://inmanga.com/thumbnails/manga/...">`, synopsis
  from a `.synopsis` div with a panel-body fallback, status text by
  walking back from the " Estado" label to the previous `<span>`.
- **getChapters** parses the chapter list and **re-sorts ascending by
  `Number`** because InManga returns chapters in lexical order
  (`1, 10, 11, …, 2, 20, …`). Without the sort the composite layer
  would surface a broken sequence.
- **getChapterPages** scrapes `<select id="PageList">` options for page
  UUIDs and builds the CDN URLs locally. Index is `option` order =
  render order. Fractional chapter numbers (`12.5`) are emitted into
  the URL verbatim.
- **getLatestUpdates**: empty success per contract — no
  unauthenticated latest-feed endpoint.
- **Mirror rotation**: defaults to single-entry list of
  `https://inmanga.com/`, but the rotator stays in place so the
  per-plugin URL override from S2.C still works.
- **App wiring**: `mangaSourcePluginsProvider` now returns
  `[MangaDex, Olympus, InManga]`. The UUID-based `sourceMangaId`
  format is consistent with how the composite repo dedupes by
  `(plugin, sourceId)` tuples.

### Residual risk

- **Detail HTML scrape is the most fragile part.** Anchor-based slicing
  works on today's markup; if InManga rebuilds with different DOM
  shapes (rare — the site has been stable for years), the parser will
  fail-soft to `inmanga.detail_not_parseable` rather than crash.
  Migrating to a proper HTML parser would harden this; deferred until
  another slice needs the same dependency.
- **Status mapping is text-only.** Localized strings like
  "En emisión" / "Finalizado" / "Cancelado" / "Pausado" are
  case-folded and prefix-matched. Numeric `BroadcastStatus` codes from
  the search JSON would be more stable but aren't returned on the
  detail page — and using them would require a search round-trip per
  detail call. Acceptable trade-off.
- **Lexical chapter order.** The plugin re-sorts ascending by parsed
  `Number`. If InManga ever ships a chapter with a non-numeric `Number`
  (none today), it will be silently dropped. The `_parseChapter`
  function returns `null` for that case to avoid surfacing garbage.
- **Mirror rotation default is single-entry.** InManga has no public
  mirrors. If they add one, it can be plumbed via the Settings UI
  override or by extending the default mirror list in a follow-up.
