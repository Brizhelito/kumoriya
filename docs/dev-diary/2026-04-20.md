## kumoriya_exoplayer — Fase 1 playback core

Construcción del motor mínimo: ExoPlayer real, Surface bindeada a la
SurfaceTexture de Fase 0, API imperativa y stream de eventos tipado.

### Cambios

- **Kotlin `PlayerInstance`**: ahora construye `ExoPlayer` con
  `DefaultMediaSourceFactory + DefaultHttpDataSource`, lo engancha al
  `Surface` creado sobre `textureEntry.surfaceTexture()` y adhiere un
  `Player.Listener` para emitir `playing`, `buffering`, `duration`,
  `completed` y `error` por el `EventChannel`
  `dev.kumoriya.exoplayer/events/<id>`. Una posting runnable en el
  main handler empuja `position` cada 200 ms mientras no haya dispose.
- **Kotlin plugin**: MethodChannel ahora acepta `open`, `play`, `pause`,
  `seek`, `setVolume`, `setSpeed` y los ejecuta siempre en main thread
  vía `Looper.getMainLooper()`. Helpers `requireLong`, `requireDouble`,
  `requireString` homogenizan el parseo de argumentos.
- **Gradle**: `media3-exoplayer-hls` y `media3-exoplayer-dash` @ 1.4.1
  sumados para HLS/DASH auto-detect sin tocar gradle en cada fase.
- **Dart**: `KumoriyaExoPlayerPlatform` extendido con la API completa y
  un `events(textureId)` que decodifica payloads con
  `PlaybackEvent.tryParse` (parser defensivo que descarta eventos
  desconocidos en lugar de crashear).
- **Controller Dart**: cachea último `playing/buffering/position/duration`,
  expone streams broadcast, protege llamadas post-dispose con
  `StateError`, cancela la suscripción al event channel antes de cerrar
  los controllers.

### Validación

- `dart analyze packages/kumoriya_exoplayer` → `No issues found!`.
- `flutter test packages/kumoriya_exoplayer/test/controller_test.dart`
  → 4 tests verdes (forwarding, stream decoding, state cache, dispose
  guards, `PlaybackEvent.tryParse`).
- Runtime Android: verificado en moto g72. El playground
  `KumoriyaExoPlayerPlaygroundPage` (debug · settings) ejecuta el flujo
  real source → `ResolverRegistry` → `controller.open(url, headers)` y
  reproduce sin regresión contra **Zilla**, **AnimeNexus** y
  **Streamwish** (más los presets públicos MP4/HLS/DASH). Eventos
  `playing` / `buffering` / `duration` / `position` / `completed` /
  `error` llegan y el render se ve sobre el `Texture` widget. Gate de
  Fase 1 ✅.

### Riesgos conocidos

- `setDefaultRequestProperties` en `DefaultHttpDataSource.Factory` se
  mueve antes del `prepare()` pero es mutable; si se llama `open()` con
  headers distintos en un mismo `PlayerInstance`, los headers nuevos
  reemplazan los viejos (buen comportamiento para Fase 2).
- Position polling a 200 ms es suficiente para UI, pero la Fase 8 (drift
  watch party) necesitará granularidad menor — revisar entonces.
- No hay visual gate todavía (`firstFrameRendered`); Fase 1 lo delegará
  al orchestrator de la app hasta que Fase 5 agregue métricas reales.

## kumoriya_exoplayer — Fase 2 cerrada (proxy killer)

Port completo del pipeline anime.nexus a Kotlin. Reproduce contra
`anime.nexus` real en moto g72 **sin** el proxy HTTP Dart. El gate de
Fase 2 queda cerrado.

### Qué entró

Módulo nuevo `dev.kumoriya.exoplayer.nexus/`:

- `NexusBrowserSession` (fingerprint UUIDv4 + cookie `sid` sintético),
  `NexusCookieJar` (OkHttp `CookieJar` con seed por host), `NexusHttpClient`.
- `NexusPageScraper` → scrapea `/watch/<uuid>/<slug>` y extrae
  `episodeId` + `attestRef` del HTML inline.
- `NexusStreamDataFetcher` → tres hops de bootstrap y `stream_data` →
  `hlsUrl` + `videoId`.
- `NexusWsClient` → Socket.IO v4 sobre OkHttp WebSocket (engine.io
  handshake + namespace ACK + auth ACK + `getInitialManifestToken` +
  `getManifestToken` + `getSegmentToken`).
- `NexusPlaybackSession` → orquesta bootstrap (off-main) y expone
  `getManifestToken` / `getSegmentToken` para el DataSource.
- `NexusDataSource` + `NexusDataSourceFactory` → Media3 `BaseDataSource`
  que firma cada request contra el WS.
- `PlayerInstance.bootstrapNexusSession` / `attachNexusSession` — split
  IO / main para respetar el threading de ExoPlayer.
- Plugin: `openNexus(textureId, watchUrl)` con `ioExecutor` single-thread
  y hop a main handler.
- Dart: `KumoriyaExoPlayerController.openAnimeNexus(watchUrl)` +
  `logStream` + evento `NativeLog`.
- Playground: sección **"anime.nexus nativo (Fase 2)"** con input de
  watch URL, botón `open vía nativo`, controles play/pause/seek y event
  log en vivo.

### Decisiones

- **Port completo en vez de tunel HTTP.** La alternativa barata (keep
  proxy Dart + override HttpDataSource en Media3) habría costado menos
  código pero perpetuaba el socket server en el proceso Flutter y el
  round-trip JSON entre isolates. Elegimos port nativo: 1 solo
  `OkHttpClient`, 0 sockets loopback, cookies compartidas con el WS.
- **WS como autoridad de firma.** En lugar de replicar la lógica HMAC
  del servidor, el DataSource pide un token nuevo por manifest/segment
  exactamente como hace el resolver Dart. Mismo surface API, distinto
  transporte.
- **Clasificador de URL estrecho.** Solo firmamos hosts `*.anime.nexus`
  y `*.cdn.nexus`. Cualquier otro host explota como
  `NexusTransportException` en lugar de leakear headers/cookies a un
  destino hostil.

### Runtime

Verificado contra `anime.nexus/watch/998f5331-08b1-4e59-b3f9-6b04c4dd99f9/...`
en moto g72. Flow observado en logcat (`NexusSession:V NexusDataSource:V`):

```
bootstrap fingerprint=04c15996-…
scrape episodeId=998f5331-… attestRef=29db8a37…
stream videoId=0199b65e-… hls=https://api.anime.nexus/.../video.m3u8
ws-open code=101
ws-ready sessionId=fd427e00942b00b09cf6e7b20772a886
open kind=MASTER uri=https://api.anime.nexus/.../video.m3u8
open kind=VARIANT_MANIFEST uri=https://cl1.cdn.nexus/.../mkv_1600-3.m3u8
open kind=INIT            uri=…_1600_init-3.mp4
open kind=SEGMENT         uri=…_1600_0000-3.m4s
```

Open total ~5 s (scrape + stream_data + WS handshake + initial token).
ABR arranca en 1600 y sube a 5300 al estabilizar bandwidth. Los
`InterruptedIOException` que aparecen al switchear calidad son cancels
de loader de Media3 — benignos, degradados a `Log.i`.

### Bugs fixados en el camino

- `classify()` chequeaba `.cdn.nexus` OR `.anime.nexus` en el orden
  equivocado; primero lo empujé a `.anime.nexus` solo y los variants en
  `cl1.cdn.nexus` caían en UNKNOWN → `ERROR_CODE_IO_NETWORK_CONNECTION_FAILED`
  ("Source error"). Fixado: acepta ambas familias.
- Regex de segmento era `_(\d+)_(\d{4})-(\d+)\.(m4s|mp4)$` — matcheaba
  el init `.mp4` y lo intentaba firmar con `getSegmentToken` (que
  necesita segmentIndex numérico). Partido en `signInit` (getManifest +
  requestType=segment) y `signSegment` (getSegment + requestType=segment).
- Query string de firma tenía un `segmentPath` espurio que el proxy
  Dart nunca envía. Removido.
- Kotlin raw string `"""episode:\$R…"""` — `$` se interpola siempre en
  triple-quoted; fix con `${'$'}R`.

### Riesgos conocidos

- **CDN edge rotation pendiente.** El proxy Dart tiene
  `NexusCdnEdgeSelector` + fallback multi-host; en Kotlin usamos
  siempre el host del master. Si ese edge se cae el stream muere. A
  agendar en Fase 8 (URL refresh + drift) o antes si aparece en
  telemetría.
- **Prefetch y cache de init/segments** (que el proxy Dart hacía con
  `_initSegmentCache` + `_mediaSegmentCache`) no están portados. Media3
  tiene su propio buffer; esperamos que el cold start sea aceptable,
  pero medir startup antes de borrar el proxy Dart.
- **Manifest refresh on auth failure.** El proxy Dart tiene
  `refreshRuntime` + circuit breaker cuando la sesión expira; en
  Kotlin, si el WS se cae la sesión queda muerta hasta que el user
  recargue. Agendar follow-up en Fase 8.
- **Borrado del proxy Dart** (`playback_proxy_server.dart`,
  `signed_hls_builder`, `playback_session_worker`, friends, ~5000 LOC)
  queda como siguiente commit tras validar unos streams más en
  producción.

## Sesión tarde — hardening del native engine y rollback a legacy

### Contexto

Tras cerrar Fase 2 por la mañana, se intentó promover
`KumoriyaExoPlayerEngine` como default en Android
(`playback_engine_factory.dart`). El flow real empezó a fallar en
varias dimensiones simultáneas y se decidió rollback inmediato para
estabilizar la app. El engine nativo queda vivo detrás de los
playgrounds.

### Cambios aplicados (todos en disco, hot-reload-safe donde aplica)

- **Rollback del routing**
  `@apps/kumoriya_app/lib/src/features/player/infrastructure/playback_engine_factory.dart:45-47`
  vuelve a `ExoPlayerPlaybackEngine()` en Android. Desktop sigue en
  `MediaKitPlaybackEngine`.
- **Rollback del registry global**
  `@apps/kumoriya_app/lib/src/features/anime_catalog/application/services/plugin_runtime_catalog.dart:34-38`
  vuelve al listado pre-bypass. `NativeAnimeNexusBypassResolver`
  queda como código huérfano accesible solo desde playgrounds
  (código + test 100 % verdes).
- **Paridad HTTP con `video_player_android`** en
  `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/PlayerInstance.kt`:
  - `setUserAgent("ExoPlayer")` en la construcción del factory.
  - En `open()`: si los headers traen `User-Agent`, se promueve al
    campo `userAgent` del factory (Media3 lo override-a sobre
    `defaultRequestProperties`). El resto de headers va a
    `defaultRequestProperties` sin la UA para evitar doble asignación.
  - Se removió `withDefaultUserAgent(...)` (ya no hacía falta).
  - `emitLog(...)` ahora espeja a `android.util.Log.i(LOG_TAG, ...)`
    con `LOG_TAG="KumoriyaExoPlayer"`, para diagnósticos offline con
    `adb logcat -s KumoriyaExoPlayer`.
- **Playground timeout** del Player Flow subido a 20 s
  (`@apps/kumoriya_app/lib/src/features/settings/presentation/pages/player_flow_playground_page.dart:180`)
  para ver qué pasa entre los 5 s y los 20 s cuando un stream tarda
  más que el SLA del engine nativo.
- **Debug log en el engine legacy**: `ExoPlayerPlaybackEngine.open`
  ahora hace `debugPrint('[exoplayer] open host=... url=...')` en
  `kDebugMode` para correlacionar `UnknownHostException` con la URL
  real sin tener que parsear el stack de Media3.

### Regresiones observadas al promover el native engine

1. **Aspect ratio roto.** `PlayerVideoSurface` renderiza el engine
   nativo como `Texture(textureId: id)` sin envolver en `AspectRatio`.
   El legacy usa `AspectRatio(aspectRatio: controller.value.aspectRatio,
   child: VideoPlayer(controller))`, por eso sí respeta la relación.
   Mitigación de hoy = rollback. Fix real queda en el gate § 11 del
   plan (exponer `videoSizeStream` desde el plugin, consumir en el
   `Texture`).
2. **Desu/Magi timeouts en real + playground.** Bajo el
   kumoriya_exoplayer engine, open > 5 s con `IO_NETWORK_CONNECTION_
   TIMEOUT`. Tras subir a 20 s salió 403 `InvalidResponseCodeException`.
   Bajo el engine legacy salió `UnknownHostException (no network)`.
3. **Hipótesis inicial (UA) → descartada.** Se probó UA Chrome-like,
   luego `ExoPlayerLib/1.9.2`, luego `"ExoPlayer"` plano (literal de
   `video_player_android`). Ningún UA mueve la aguja porque el gate
   no es por UA.

### Diagnóstico externo del CDN de JKAnime

Se validó desde la laptop y desde el phone que el problema es del
proveedor, no del cliente:

| Test | Resultado |
|---|---|
| `dig nika.playmudos.com A` | `104.21.89.108`, `172.67.140.136` (Cloudflare) |
| `ping` desde phone | ✅ 44 ms |
| TLS 1.3 + HTTP/2 handshake desde laptop | ✅ cert `*.playmudos.com`, HTTP/2 OK |

es un bug de nuestro stack de red.

Usuario confirmó independiente: "JKanime anda lento en general, solo
carga rápido la pagina de home". Estado del proveedor.

## Sesión noche — anime.nexus WiFi 403 "Token validation failed"

### Contexto

Reproduciendo anime.nexus en WiFi del ISP doméstico, el native engine
devolvía sistemáticamente `403 {"error":"Token validation failed"}` en
cada `VARIANT_MANIFEST`. En datos móviles y con VPN (desde la misma
WiFi) reproducía limpio. Mismo código, misma session, mismo fingerprint.

### Diagnóstico

Comparando el netlog real del SPA web (captura Chrome
`/tmp/nexus-netlog.json.gz` → `extract_nexus_netlog.py` →
`/tmp/nexus-slim.json`) contra el wire de nuestro port Kotlin,
aparecieron tres deltas: falta de `sec-ch-ua` trio + `cache-control:
max-age=0` en los headers, y falta del query param `segmentPath=<path>`
en `signSegment` / `signInit`. Se implementaron los tres fixes y en la
primera prueba real los 403 persistieron. Luego el usuario probó VPN
manteniendo el resto idéntico y reprodujo al toque — **confirmando que
la raíz es reputación por IP del ISP en el WAF de Cloudflare/edge, no
un delta de cliente**. Los tres cambios fueron revertidos (ver
`NexusConstants.kt`, `NexusPlaybackSession.browserFetchHeaders`,
`NexusDataSource.signInit/signSegment`).

Hipótesis previas descartadas en orden: timing (35 s de `fetchHealth`
en WiFi era síntoma, no causa, porque el bug predata al edge-selector),
cookie scoping (`anime_nexus_session` solo en `.anime.nexus`, no llega
al CDN — idéntico en ambos transportes), sec-ch-ua (tras probar, no
cambia nada en esa IP), pinning WS↔CDN (edge-selector elige `us1`
consistentemente pero el server la rechaza global).

### Cambios

Defensa en profundidad contra el mismo failure mode cuando llegue como
bug per-edge en el futuro (p.ej. un edge entra en blacklist transitoria
mientras el resto está limpio):

- `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/nexus/NexusEdgeSelector.kt`
  gana `selectRanked()` que devuelve `Selection(primary, fallbacks)`
  con todos los edges que respondieron al ping ordenados por RTT. El
  `select()` legacy sigue vivo como wrapper. Se agrega
  `invalidateCache()` para limpiar el TTL cuando la sesión agota la
  lista.
- `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/nexus/NexusPlaybackSession.kt`
  hace mutable el `edgeHost` (via `edgeHostRef` volátil + getter) y
  guarda `edgeFallbacks: ArrayDeque<String>`. Método
  `rotateEdge(reason)` popea el siguiente candidato, loggea y devuelve
  false cuando se agotaron — ahí invalida la caché global. Además
  tracking de `burnedEdges: Set<String>` vía `recordTokenRejection` +
  `isLikelyWafBlocked(threshold = 2)` para cortar antes.
- `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/nexus/NexusDataSource.kt`
  — `open()` ahora es un loop con presupuesto de una rotación por call.
  En 403 con body `"Token validation failed"` registra el edge quemado;
  si la session ya tiene ≥2 edges quemados, tira
  `NexusBlockedByWafException` envuelto en `IOException` + `HttpDataSourceException`
  en vez de seguir rotando (rotar cuesta ~1 s por edge y nunca recupera
  cuando el IP está baneado globalmente). Master manifests (que viven
  en `api.anime.nexus`) nunca disparan rotación.
- `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/nexus/NexusErrors.kt`
  — nueva `NexusBlockedByWafException(message, cause?)` para distinguir
  este failure mode de un 403 transitorio.

### Validación runtime (moto g72 en WiFi "mala")

Log `kxplayer_log_20260420_171135.txt`: la rotación corrió como se
esperaba y probó 7 edges (us1 → us3 → ca1 → eu1 → cl1 → br1 → us5)
antes de que Media3 agotara su retry budget. Cada uno devolvió 403
con el mismo body — confirmando que la detección `isLikelyWafBlocked`
se activaría al segundo edge (us3) y cortaría en ~2 s en vez de los
~15 s observados, una vez que el fail-fast se builde en device.

### Pendiente

- [ ] Rebuild + re-test del fail-fast en el mismo WiFi para validar
      que el `NexusBlockedByWafException` corta en 2 edges quemados.
- [ ] UI mapping: cuando el error llegue al orchestrator Dart,
      desempacar la cause y mostrar "Tu red está bloqueada, probá
      VPN" en lugar del `ERROR_CODE_IO_BAD_HTTP_STATUS` genérico.
      Slice separado, probablemente junto a Fase 8 (error taxonomy).

## Sesión noche — aspect ratio del engine nativo

### Contexto

Primer ítem del gate §11 del plan. El `Texture` widget del native
engine en `@apps/kumoriya_app/lib/src/features/player/presentation/widgets/player_video_surface.dart`
no estaba envuelto en `AspectRatio`, así que el video salía estirado
al `BoxFit` del padre. El legacy `video_player` branch sí lo envuelve
usando `VideoPlayerController.value.aspectRatio`. Había que replicar
eso sin acoplarnos al `video_player`: el plugin nativo tenía que

- `@packages/kumoriya_exoplayer/android/src/main/kotlin/dev/kumoriya/exoplayer/PlayerInstance.kt`
  implementa `Player.Listener#onVideoSizeChanged` y emite el evento
  `videoSize` con `width` pre-multiplicado por `pixelWidthHeightRatio`
  (así Dart trata la pareja como "display-ready" y calcula
  `width / height` sin lidiar con pixeles anamórficos). Misma
  pre-multiplicación en el snapshot que se manda al re-suscribirse,
  para no depender del próximo cambio para listeners tardíos.
- `@packages/kumoriya_exoplayer/lib/src/events/playback_event.dart`
  — nueva variante `VideoSizeChanged(width, height)` con getter
  `aspectRatio` (fallback 16/9 cuando `height <= 0`).
- `@packages/kumoriya_exoplayer/lib/src/kumoriya_exoplayer_controller.dart`
  expone `videoSizeStream` + caché `videoSize` para late subscribers,
  cierra el nuevo stream en `dispose()`.
- `@apps/kumoriya_app/lib/src/features/player/infrastructure/kumoriya_exoplayer_engine.dart`
  gana `ValueNotifier<double?> aspectRatio`, se resetea en cada
  `open()`, se dispara desde `videoSizeStream`, y se libera en
  `dispose()`.
- `@apps/kumoriya_app/lib/src/features/player/presentation/widgets/player_video_surface.dart`
  wrappea el `Texture` en `ValueListenableBuilder` → `AspectRatio`.
  Fallback a 16:9 hasta que llega el primer `onVideoSizeChanged`
  (mismo comportamiento que el legacy durante init).

## Sesión 5 — Fase 3: audio tracks embebidos (extremo a extremo)

Con el aspect ratio cerrado del lado del código, avanzamos Fase 3 hasta
dejar toda la plomería lista. Ya estaban en Kotlin el `onTracksChanged`
que dispara `emitAudioTracksEvent` + el método `selectAudioTrack` con
`TrackSelectionParameters.setOverrideForType`; hoy cerramos el lado
Dart y el engine de la app.

### Dart plugin (`packages/kumoriya_exoplayer`)

- **`AudioTrack`** model (`lib/src/models/audio_track.dart`) con
  `id` (opaco, `groupIndex:trackIndex`), `label`, `language`, `codec`,
  `channels`, `sampleRate`, `bitrate`, `selected`. Expuesto desde
  `kumoriya_exoplayer.dart`. `tryParse` descarta entradas sin id.

### Fase 3b — subtítulos embebidos + externos

Kotlin (`PlayerInstance.kt`):

- `selectSubtitleTrack(trackId)` y `clearSubtitleTrack()` — mismo
  esquema que el audio track (TrackSelectionOverride +
  `setTrackTypeDisabled` para habilitar/deshabilitar el tipo).

  `MergingMediaSource(base + SingleSampleMediaSource...)` preservando
  posición vía `player.currentPosition`. MIME normalizado para
  `text/vtt`, `application/x-subrip`, `text/x-ssa` (alias incluidos).
- `emitSubtitleTracksEvent(tracks)` en `onTracksChanged` + snapshot.
  Se emite siempre, incluso con lista vacía, para que Dart pueda
  limpiar su cache cuando el stream nuevo no tiene subs.
- Internos: `currentBaseSource: MediaSource?`, `externalSubtitles:
  MutableList<ExternalSubtitleSpec>`, helpers `installBaseSource`,
  `rebuildWithExternalSubtitles`, `buildSourceWithSubtitles`
  (`treatLoadErrorsAsEndOfStream=true` para que un CDN de subs flaky
  no fume la playback entera).

Dart:

- Nuevo `SubtitleTrack` model + `SubtitleTracksChanged` event.
- `KumoriyaExoPlayerController`: `subtitleTracks`,
  `subtitleTracksStream`, `selectSubtitleTrack`, `clearSubtitleTrack`,

- Platform + MethodChannel impls para los cuatro métodos nuevos.

  `clearSubtitleTrack()` ya no son no-op — forward al controller.

  `setSubtitleTrack` detecta `track.uri == null` (caso inline `data`)
  y loguea que no está soportado — queda como follow-up slice.
- `_publishEmbedded()` ahora publica un `EmbeddedTracks` combinado
  (audio + subtitle) en cada cambio, reemplazando los dos listeners
  separados del commit anterior. Evita carreras donde un stream
  multi-pista emite dos `EmbeddedTracks` consecutivos sobrescribiendo

### Fase 8 — URL refresh interceptor

Kotlin:

- `currentBaseUrl: String?` tracking en `open()` (null en
  `attachNexusSession` — Nexus firma por-request).
- `onPlayerError` walk el cause chain con
  `generateSequence`/`filterIsInstance<InvalidResponseCodeException>`
  buscando 401/403/410. Si hay match + `currentBaseUrl` != null,
  emite `urlExpired { url, httpCode }` **antes** del `error` genérico
  para dar chance de recovery silencioso.
- `swapUrl(url, headers, mimeType, startPositionMs=null)`: rebuild
  del base source con los headers nuevos, conserva
  `externalSubtitles` (el usuario no pidió tirarlos, solo cambió el
  stream), y seek-preserva con `player.currentPosition` fallback.

Dart:

- Nuevo `UrlExpired` event + decode branch.
- `controller.urlExpiredStream` + `controller.swapUrl(...)`.
- `KumoriyaExoPlayerEngine`: typedef `UrlExpiredResolver` +
  `RefreshedStreamUrl` record + constructor param `onUrlExpired`.
  Handler `_handleUrlExpired` llama al hook, swallowea errores,
  si devuelve nueva URL hace `controller.swapUrl`. Cuando `onUrlExpired`
  es `null` solo loguea — el flag queda off hasta que el orchestrator
  lo cablee.

Tests (2 nuevos):

- `PlaybackEvent.tryParse decodes urlExpired payload`.
- `controller fans out UrlExpired and forwards swapUrl`.

Deferral explícito: wiring con el `ResolverRegistry`/orchestrator
real (slice aparte porque requiere tocar `player_session_orchestrator`
y coordinar con `NativeAnimeNexusBypassResolver`). Speed ramp
(`applySpeedRamp` para drift de watch party) tampoco entra — el plan
lo describe como loop con feedback, que es slice propia.

### Fase 4 — audio boost voz

Kotlin:

- `LoudnessEnhancer` creado/destruido en base a `pendingGainDb`.
  Negativos clamp a 0 (AudioFX no soporta atenuación por este
  camino; el usuario debe usar `setVolume` para bajar).
- `DynamicsProcessing` (API 28+) con preset 3-band: cut -6 dB sub-120
  Hz, boost +4 dB @1.5 kHz, shelf flat @6 kHz, todo escalado por
  `strength ∈ [0,1]`. Fallback silencioso en API<28 con log claro.
- `applyPendingAudioFx()` reaplica todo en `STATE_READY` porque
  Media3 asigna `audioSessionId` tardío; mientras no hay sesión el
  setter solo cachea `pendingGainDb`/`pendingVoiceClarity`.
- Release cleanup: `releaseAudioFx()` en `release()`.

Dart:

- `controller.setOverallGainDb(db)` + `controller.setVoiceClarity(s)`.
- Platform + MethodChannel.
- Engine `setSmartAudioBoost(enabled)` ya no es no-op — mapea a
  +6 dB / 0.7 clarity on, (0, 0) off. Números intencionalmente
  conservadores — tuning fino requiere A/B con oído en AnimeNexus ES.

Tests (1 nuevo): `controller forwards setOverallGainDb + setVoiceClarity`.

Deferral: gate runtime de tuning en device.

### Fase 5 — diagnostics overlay

Kotlin:

- `DiagnosticsCollector` nuevo archivo: implementa `AnalyticsListener`
  y muta campos en cada callback (`onDroppedVideoFrames`,
  `onVideoInputFormatChanged`, `onVideoDecoderInitialized`
  —detecta HW/SW por prefijo `OMX.google.` / contains `.sw.`—,
  `onAudioInputFormatChanged`, `onBandwidthEstimate`).
- `PlayerInstance.setDiagnosticsEnabled(enabled)`: añade/quita el
  listener, arranca/detiene polling @1 Hz que emite `diagnostics`
  event con snapshot completo (frame drops, codec, decoder, HW flag,
  audio info, bandwidth, buffered, video size).
- Off por default — la app solo paga el costo cuando el overlay
  está visible.

Dart:

- Nuevo `DiagnosticsSnapshot` model exportado + `DiagnosticsReport`
  event + decode branch.
- `controller.diagnosticsStream` + `controller.setDiagnosticsEnabled(enabled)`.
- Platform + MethodChannel.
- **Engine lazy enable**: `_diagnostics` es un `StreamController`
  con `onListen`/`onCancel` que cuenta subscribers. Primer listener
  llama `setDiagnosticsEnabled(true)` + suscribe al stream nativo;
  último cancel → `setDiagnosticsEnabled(false)`. Si `open()` crea
  un controller mientras hay listeners activos, el attach re-habilita
  diagnostics en el controller fresco. Así el overlay debug "just
  works" across stream swaps sin pagar costo en release builds donde
  nadie subscribe.
- Mapeo del `DiagnosticsSnapshot` nativo → `PlayerDiagnostics`
  compartido con media_kit para que `PlayerDebugOverlay` no tenga
  que distinguir engine.

Tests (1 nuevo compuesto): decode payload completo + controller
fan-out + forward del `setDiagnosticsEnabled`.

Deferrals: widget overlay Media3-aware (si el existente no alcanza),
FPS real (Media3 no expone el equivalente de mpv's `estimated-vf-fps`

### Riesgos residuales (acknowledged)

El plan §11 los lista en detalle. Resumen para este diario:

1. **Aspect ratio del surface**: la fix de la sesión previa (envolver
   `Texture` en `AspectRatio` con `onVideoSizeChanged`) está en código
   pero **no validada visualmente**. Si un stream 4:3 o uno sin
   dimensiones sale estirado, rollback inmediato del factory.
2. **Desu/Magi**: nunca se smokearon con este engine en la ventana de
   esta sesión. Si Cloudflare no está challenging a nika.playmudos.com,
   debería andar (el patch de UA byte-a-byte con `video_player_android`
   se aplicó). Si hay regresión, no es por este commit sino por un
   bug del plugin plumbing.
3. **MediaFire URL expiry**: el plumbing de Fase 8 está pero
   `createPlaybackEngine` no pasa `onUrlExpired` al engine. Cualquier
   stream que caduque mid-playback va a surfacear el 403/410 como
   error terminal en vez de recover. Si los usuarios reportan "el
   video se cortó y no vuelve", es esto. Fix: wiring de Fase 8 via
   slice propia (toca `player_session_orchestrator`).
4. **Fase 3/3b/4/5 runtime gates**: plumbing testeado sí, pero
   comportamiento real en device con content real — **no**. Posibles
   sorpresas: decoder HW/SW detection false positive en OEMs raros
   (Fase 5), LoudnessEnhancer rechazado en algunos Android stocks
   (Fase 4), MergingMediaSource que corte un segundo de audio en el
   swap de external subtitles (Fase 3b).
