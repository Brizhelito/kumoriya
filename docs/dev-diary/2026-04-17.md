## 22:30 ŌĆö Player performance investigation (Linux desktop)

**Category:** fix

**Affected files:**
- `apps/kumoriya_app/lib/src/features/player/presentation/pages/player_page.dart`
- `apps/kumoriya_app/lib/src/features/player/presentation/widgets/player_debug_overlay.dart`

**Summary:**
User reported severe playback stutter on Linux desktop at 1080p despite mpv reporting stable 24 fps decode. Root cause: `media_kit_video` on Linux renders the video through a Flutter `Texture`; the texture is only composited when Flutter produces a frame. Without active animations, Flutter entered idle and composited ~5 fps, so the user saw the video at that rate regardless of mpv's decode rate. Additional contributor: three always-on `BackdropFilter(ImageFilter.blur)` layers in the controls overlay (play/pause Žā=12, skip Žā=8, speed badge Žā=8) forced `saveLayer` + gaussian blur every frame.

Fixes applied:
1. Added a continuous vsync pump (`Ticker` with empty callback) in `_ImmersivePlayerViewState`, started/stopped via `didUpdateWidget` mirroring `widget.isPlaying`. Forces Flutter to schedule a frame every vsync while playing so the video texture composites at display refresh rate.
2. Removed all three `BackdropFilter` instances from the controls overlay, replaced with flat tinted containers (opacity bumped from 0x73/0xB3 to 0xCC/0xD9 to compensate for lost blur legibility).
3. Debug overlay (`PlayerDebugOverlay`) now tracks the real Flutter compositor FPS via `SchedulerBinding.addTimingsCallback` over a rolling 2 s window, with `build` vs `raster` split and `slow>16ms` / `slow>33ms` counters. This distinguishes render lag from decode lag and is reused for future profiling.

Result on Intel HD Graphics 4600 (2013 iGPU, X11, Mesa 25): UI FPS went from ~5 to ~30 stable, `build avg = 1.9 ms`, `raster avg = 13.4 ms`, zero drops. Reached the hardware ceiling for this iGPU ŌĆö raster cost of compositing a 1080p texture in Flutter on HD 4600 is ~14 ms, near the 16.6 ms vsync budget, which caps pacing at 30 fps.

**Discarded attempts (reverted):**
- `VideoControllerConfiguration(height: 720)` cap + dart-define ŌĆö did not take effect as expected; overlay kept reporting 1080p texture and raster did not drop.
- `gpu-hwdec-interop=auto` / `hwdec-codecs=all` via `setProperty` post-init ŌĆö caused render drops and pipeline instability (fps dropped to 11). mpv ignored the interop hint anyway and kept `vaapi-copy`. Setting properties after media_kit's render API initialized re-init the decoder mid-flight.

**Residual risk / known limitation:**
- 60 fps on Linux desktop is unreachable from inside the Flutter tree on this iGPU. YouTube/VLC/mpv achieve 4K@60 via native X11/Wayland hardware overlay, bypassing the app's render pipeline. A native X11 subsurface plugin is the only path to parity; captured as a possible future epic, not part of the player slice.
- Android/iOS are expected to hit 60 fps since their texture pipelines (SurfaceTexture + HW composer) have distinct and cheaper semantics. Validation pending.

**Linked phase:** Player slice ŌĆö stability / perf hardening

## 22:55 ŌĆö Fix: watch party members auto-playing on entering episode

**Category:** fix

**Affected files:**
- `apps/kumoriya_app/lib/src/features/watch_party/application/party_session_guard.dart`
- `apps/kumoriya_app/lib/src/features/player/presentation/pages/player_page.dart`
- `apps/kumoriya_app/test/watch_party_session_guard_test.dart`

**Summary:**
Bug: when a non-host member opened an episode inside an active watch party, their player auto-played from position 0 the moment all members were marked ready, regardless of the host's actual playback state. Root cause: `_shouldHoldPartyPlayback` only gated on `partyHasAllMembersReady` ŌĆö once all ready, the hold released and `orchestrator.start()`'s auto-play was never suppressed. The server's `playback_state_changed` sync arrived after and would toggle pause, but by then frames of wrong playback had already leaked.

Fix: extracted the hold logic from `_PlayerPageState` into a pure `shouldHoldPartyPlayback({session, isLocallyBoundToRoom, isLocalHost})` in `party_session_guard.dart`, and added a second hold condition: **non-host members stay paused until the server's authoritative `PartyPlaybackState.isPlaying` flips to `true`**. Effectively members now mirror the server state at all times instead of racing `orchestrator.start()` against the first sync event. Also refined the `_togglePartyAwarePlayPause` snackbar to distinguish "waiting for everyone" from "only the host can control playback" for non-hosts.

Added 5 regression tests under `group('shouldHoldPartyPlayback')` covering: no-op when unbound, hold when any member loading, host plays post-ready, member stays paused post-ready with server paused, member releases on server-play. All 14 tests in the guard suite pass.

**Hot-fix**: first runtime attempt threw `ProviderException: Cannot use Ref or modify other providers inside life-cycles/selectors` because the delegating `_shouldHoldPartyPlayback` method calls `ref.read(partySessionProvider.notifier)` internally, and it was being invoked from a `ref.watch(partySessionProvider.select((session) => ..._shouldHoldPartyPlayback(session)))` selector inside `build` (line 1397). Fixed by calling the pure `shouldHoldPartyPlayback` directly at that call site with the locally-computed `isLocalHost` already present there, keeping the `.select` closure free of any `ref` reads. The instance method remains for the non-selector call sites (`_enforcePartyPauseHold`, `_onPlayingChanged`, `_togglePartyAwarePlayPause`, `_schedulePartyPauseHoldIfNeeded`) where `ref.read` is permitted.
