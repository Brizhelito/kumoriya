**Summary:**
First sub-slice of the manga multi-source matching plan (S1.A): introduce a leaf package that wraps `https://api.mangabaka.dev/v1/`. Reconnaissance showed the `api.` subdomain answers without Cloudflare gating (only the homepage is challenged), so no proxy infrastructure is required. The package mirrors the AniList adapter conventions: sealed error hierarchy, `Result`-based contract, defensive mapper, HTTP client with min-request gap + Retry-After honoring + in-memory TTL cache, and a high-level gateway with two ops (`searchSeries`, `fetchSeriesById`) that follows merged-row redirects once.

The real value of MangaBaka for Kumoriya is the **title corpus** (e.g. 27 known variants for Solo Leveling across languages) which will dramatically improve fuzzy matching against scanlator sources that don't expose AniList ids. Cross-tracker IDs (AniList, MAL, Kitsu, MangaUpdates, AnimePlanet, Anime News Network, Shikimori) are also exposed for the future M8 picker enrichment slice.

The package is a leaf — no app code imports it yet. Wiring into the composite repository happens in S1.D after the multi-source plumbing lands in S1.C.

**Residual risk:**
- The cover-URL extractor falls back to "first http-prefixed string under any sub-key" when the `raw.url` shape changes. If MangaBaka renames the field to something with a non-http scheme (e.g. relative paths), the cover would silently become null. Acceptable — the matching pipeline does not need cover URLs, only metadata.
- The merge-follow logic is single-hop. If MangaBaka ever produces transitive merges (`A merged_with B; B merged_with C`), we surface row B instead of C. Defensive cap is intentional to keep gateway latency bounded; if observed in the wild, lift the limit to 2-3 hops with a visited-set guard.
