**Summary:**
Second sub-slice of the manga multi-source matching plan (S1.B): introduce a leaf package that wraps `https://api.mangaupdates.com/v1/`. Mirrors the AniList and MangaBaka adapter conventions but adds a `postJson` operation alongside `getJson` because the v1 API mixes GETs (detail endpoints) with POSTed JSON bodies (search endpoints).

The package exposes a `MangaUpdatesMetadataGateway` with four ops:

  * `searchSeries(query)` — relevance-ranked list of series, used as a fallback resolver when MangaBaka does not cover a row.
  * `fetchSeriesById(id)` — full detail with `associated` titles for matching, plus `latest_chapter`, `completed`, `licensed`, and a free-form status note.
  * `fetchGroupById(id)` — canonical scanlator group surface with the `active` flag (the killer signal for picker enrichment in S1.F).
  * `searchReleases({seriesId, groupId})` — release timeline used to compute "last activity per scanlator on this series".

`MangaUpdatesSeries.titleCorpus` mirrors `MangaBakaSeries.titleCorpus` so the matching pipeline can consume both gateways uniformly.

Operational findings recorded as test guards:
- MangaUpdates `series_id` and `group_id` are 64-bit ints (e.g. 15_180_124_327). Documented in the model class doc; native Dart handles it, JS targets must avoid round-tripping through a JS number.
- MU returns **404 with an empty body** (no JSON envelope), unlike MangaBaka. Client maps status code → `MangaUpdatesNotFoundError` directly.
- Genres come back as `[{genre: "Action"}, ...]` rather than flat strings; mapper accepts either shape.
- `time_added` is parsed from the rfc3339 form preferentially with fallback to the unix timestamp.

**Summary:**
Fourth sub-slice of the manga multi-source matching plan (S1.D). The composite repository now consumes the leaf `MangaBakaMetadataGateway` (built in S1.A) as an optional second-tier matcher between the existing AniList-id direct match (Strategy A) and the legacy fuzzy fallback (Strategy B):

  * **Strategy A2 (cross-tracker bypass).** When MangaBaka has a row whose `crossIds.anilistId == manga.anilistId`, the resolver also accepts a source row whose `externalIds['mu']` aligns with `crossIds.mangaUpdatesId` or whose `externalIds['mal']` aligns with `myAnimeListId`. This is the path that closes MangaDex's "different romanization, same MU id" gap without ever invoking fuzzy matching.
  * **Strategy B+ (corpus expansion).** When fuzzy fallback runs, the candidate pool now includes MangaBaka's `titleCorpus` (canonical + romanized + native + secondary titles). Closes the "AniList only knows the Japanese romaji form, the source uses the English fan title" class of misses.
  * **Per-AniList-id memoization** on the composite, sharing a single in-flight `Future<MangaBakaSeries?>` across the parallel fan-out so N plugins cause exactly one MangaBaka call.
  * **Failure isolation.** `MangaBakaMetadataGateway` failures (transport, mapping, exceptions) are logged via `developer.log` and treated as "no MangaBaka context" — the resolver degrades to the legacy A+B path. A MangaBaka outage does not take chapter resolution down.
  * **Optional injection.** New constructor parameter `mangaBaka: MangaBakaMetadataGateway?` defaults to null. Existing call sites and tests keep their behavior unchanged.

Producing a real `HttpMangaBakaMetadataGateway` requires HTTP client + cache config + a Riverpod provider with proper teardown; that wiring lands in S1.E or S1.F alongside the source picker chip.
