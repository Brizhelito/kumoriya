**Summary:**
New pure-Dart package `kumoriya_manga_plugins`, parallel to `kumoriya_plugins`. Defines:

- `MangaSourcePlugin` — async contract over `Result<T, KumoriyaError>` with `search`, `getLatestUpdates`, `getMangaDetail`, `getChapters`, `getChapterPages`. Plugins that don't support a feature (e.g. latest feed) MUST return an empty `Success`, not a failure.
- Source-side models: `SourceMangaMatch` (light search row, AniList alignment fields), `SourceMangaDetail` (synopsis/authors/artists/tags/status/originalLanguage), `SourceChapter` (`double number` for 12.5/side-stories, optional `volume`/`language`/`scanlator`/`publishedAt`/`pageCount`), `SourcePage` (`index >= 0`, direct `imageUrl`, optional per-request `headers`, optional `width`/`height`).
- Request value objects: `MangaSearchQuery` and `MangaChapterQuery` with 1-indexed pages and asserted invariants.
- `MangaSourceCapabilities` — a separate value type carrying per-plugin feature flags (`supportsLanguageFilter`, `supportsScanlatorFilter`, `supportsLatestFeed`, `requiresPageHeaders`). Kept off `PluginCapability` to avoid polluting the anime-flavored enum; manga flags are decision-axes anime sources never need.

Reuses `PluginManifest` and `PluginType` from `kumoriya_plugins` as-is — no fork. Depends only on `kumoriya_core`, `kumoriya_manga_domain`, `kumoriya_plugins`. Zero coupling to a concrete source, the reader, or storage.

Workspace `pubspec.yaml` registers the new package. Tests cover capability defaults, value equality, query-invariant asserts, fractional chapter numbers, and a fake plugin satisfying the full interface. `dart format` clean, `dart analyze` 0 issues on the new package and across the workspace, 8/8 contract tests pass.

**Residual risk:** the package is dead code until Slice 6 (`kumoriya_source_mangadex`). The contract surface — particularly chapter `language`/`scanlator` shape and `SourcePage.headers` — is the most likely thing to need a reshape once a real source hits it; intentionally kept minimal so that reshape is cheap.

**Summary:**
Schema bump 19 → 20 with seven strictly-additive tables for the manga universe: `manga_cache`, `manga_chapter`, `manga_progress`, `manga_history`, `manga_library`, `chapter_page_cache`, `manga_download`. Anime tables and DAOs untouched (asserted by the new v19→v20 regression test). Each table has a typed Drift definition (`REAL` chapter numbers preserve `12.5`/side-stories, `Map<String,String>` HTTP headers serialised as JSON via `dart:convert`), a DAO, a contract returning `Result<T, KumoriyaError>`, and a `Drift*Store` implementation. Mirrors the existing anime pattern (`AnilistCacheStore`/`LibraryStore`/`AnimeProgressStore`/`DownloadStore`) with manga-specific differences: `MangaCacheStore` carries `countryOfOrigin`, `originalLanguage`, `totalChapters`, `totalVolumes`; `MangaChapterCacheStore` is keyed on `(sourceId, sourceChapterId)` so the same chapter from a different scanlator/language is a different row; `ChapterPageCacheStore` indexes the on-disk page cache (bytes live on disk, this table holds metadata + TTL); `MangaDownloadStore` adds a `partial` status for chapters that downloaded page-by-page but failed to package into a CBZ.

`MangaCacheStore.upsert` mirrors the anime cache trick: when `relationsJson == null` the column is left untouched (`Value.absent()`) so list-level refreshes don't NULL-out relations cached by an earlier detail-level fetch. Library "soft-delete" mirrors the anime pattern too: removing a favorite leaves the row when `notify`/`autoDownload`/`preferredLanguage`/`preferredScanlator` are still set.

Eleven new partial/composite indices, all prefixed `idx_manga_*` to avoid collisions. Migration step at `from < 20` invokes `_createTableIfMissing` per table and `_createMangaIndices()`. `beforeOpen` for fresh DBs now also calls `_createMangaIndices()`.

**Summary:**
First real `MangaSourcePlugin` implementation, against MangaDex's public REST API. Entire surface implemented: `search`, `getLatestUpdates`, `getMangaDetail`, `getChapters`, `getChapterPages`. All five methods return `Result<T, KumoriyaError>`; the plugin never throws across the contract boundary.

Endpoints used:

- `GET /manga?title=…&limit=&offset=&availableTranslatedLanguage[]=&includes[]=cover_art` — search.
- `GET /manga?order[latestUploadedChapter]=desc&hasAvailableChapters=true&includes[]=cover_art` — latest updates.
- `GET /manga/{id}?includes[]=cover_art&includes[]=author&includes[]=artist` — detail.
- `GET /manga/{id}/feed?translatedLanguage[]=…&order[chapter]=asc&order[volume]=asc&includes[]=scanlation_group` — chapter feed.
- `GET /at-home/server/{chapterId}` — page bundle. Image URLs are built as `{baseUrl}/data/{hash}/{filename}` per the MD@Home contract; non-data-saver bucket.

Capabilities advertised: `supportsLanguageFilter=true`, `supportsScanlatorFilter=true`, `supportsLatestFeed=true`, `requiresPageHeaders=false` (MangaDex's MD@Home does not require Referer/Origin pinning for image fetches).

Parser quirks worth noting:

- MangaDex titles are `Map<String, String>` localized objects; the plugin picks the first available value as primary and stuffs the rest plus `altTitles` into `aliases`.
- `MangaFormat` is inferred from `originalLanguage` plus the `Long Strip` / `Web Comic` / `Oneshot` / `Doujinshi` tag set, since MangaDex does not expose a single discriminator equivalent to AniList's `format`.
- `status` maps `ongoing→releasing`, `completed→finished`, `hiatus→hiatus`, `cancelled→cancelled`; anything else falls to `unknown`.
- Chapters with `chapter == null` (unnumbered prologues) are filtered out, since they cannot be ordered or addressed by `MangaChapterProgress.chapterNumber`. May revisit if a UX use case appears.
- Scanlator filter is post-applied because MangaDex's feed endpoint does not accept a scanlation-group filter directly.

HTTP envelope handling: `result == "ok"` is required; anything else maps to `mangadex.api_error`. 4xx/5xx map to `notFound` / `transport`. JSON decoding failures map to `mapping`. Search/list parse failures wrap in `mangadex.*_parse_failed` mapping errors.

**Verification:** `dart format` clean, `dart analyze` 0 issues at package and workspace level, 16/16 plugin tests pass. Tests use `package:http/testing.dart` `MockClient` against recorded JSON fixtures (no live network).

**Test infra detail worth remembering:** `http.Response(body, 200)` defaults to Latin-1 encoding when no `Content-Type` charset is set, which crashes the request-side validation when fixtures contain non-ASCII (e.g. `チェンソーマン`). The test file wraps responses in a small `_ok(body)` helper that sets `content-type: application/json; charset=utf-8`. Future manga-source tests should follow the same pattern.

**Workspace:** registered the new package in the root `pubspec.yaml` workspace list.

**Residual risk:** the plugin is not yet wired to any repository — it's verified only via fixture-backed unit tests. First real integration happens in Slice 8 (Manga Home/Search/Detail) where `MangaCatalogRepository` will compose AniList + matching + this plugin. The two areas most likely to need iteration are (a) MangaDex rate limiting (no per-request budget yet; will be added once the repository layer hits real load) and (b) the `_parseFormat` heuristic, which is best-effort given MangaDex's tag-based format model.

**Summary:**
The flat `KumoriyaTab` enum is split into `KumoriyaAnimeTab { home, search, calendar, library, downloads }` (5 tabs, unchanged from before) and `KumoriyaMangaTab { home, search, library, downloads }` (4 tabs — no Calendar, since Calendar is anime-only per plan §8, and no dedicated Latest tab since Latest lives inside Manga Home).

`AppNavigationShell` becomes universe-aware: it watches a new `activeUniverseProvider` (Riverpod `Notifier<MediaKind>`) and rebuilds its bottom nav / desktop rail from the active universe's tab spec. Each universe keeps its own `_currentTabIndex`, `_visitedTabs` set, and per-tab `GlobalKey<NavigatorState>`, so flipping universes does not destroy the inactive universe's navigation stacks. The shell takes two new parameters — `animeTabBuilders` and `mangaTabBuilders` — and the old single-map `tabBuilders` is gone.

Active universe state is persisted via `ActiveUniverseStore`, a tiny file-based singleton that mirrors the existing `SeenAppVersionStore` pattern (one text file in the application support directory, storing `MediaKind.wireValue`). The store ships with a `forTesting` constructor that injects a directory resolver, so widget tests don't need `path_provider_platform_interface` mocking — they just point the store at a `Directory.systemTemp.createTemp` fixture.

The `ActiveUniverseNotifier` defaults to `MediaKind.anime` and kicks off a `Future.microtask` to load from disk on first build; this is the cheapest way to keep the UI from blocking on a disk read at first paint. Every set is persisted fire-and-forget.

`UniverseSwitch` is a small two-segment pill (Anime / Manga). It's a `ConsumerWidget` that reads + writes `activeUniverseProvider` directly. Per the user's design call, it lives **only** in the Home page header of each universe — anime `_MobileHeader` (in `home_page.dart`) and the new `MangaHomePage`. Other tabs inherit the universe silently. Two taps to switch from Library/Downloads (Home → switch), accepted trade-off.
