**Summary:**
First real `MangaSourcePlugin` implementation, against MangaDex's public REST API. Entire surface implemented: `search`, `getLatestUpdates`, `getMangaDetail`, `getChapters`, `getChapterPages`. All five methods return `Result<T, KumoriyaError>`; the plugin never throws across the contract boundary.

Endpoints used:

- `GET /manga?title=‚Ä¶&limit=&offset=&availableTranslatedLanguage[]=&includes[]=cover_art` ‚Äî search.
- `GET /manga?order[latestUploadedChapter]=desc&hasAvailableChapters=true&includes[]=cover_art` ‚Äî latest updates.
- `GET /manga/{id}?includes[]=cover_art&includes[]=author&includes[]=artist` ‚Äî detail.
- `GET /manga/{id}/feed?translatedLanguage[]=‚Ä¶&order[chapter]=asc&order[volume]=asc&includes[]=scanlation_group` ‚Äî chapter feed.
- `GET /at-home/server/{chapterId}` ‚Äî page bundle. Image URLs are built as `{baseUrl}/data/{hash}/{filename}` per the MD@Home contract; non-data-saver bucket.

Capabilities advertised: `supportsLanguageFilter=true`, `supportsScanlatorFilter=true`, `supportsLatestFeed=true`, `requiresPageHeaders=false` (MangaDex's MD@Home does not require Referer/Origin pinning for image fetches).

Parser quirks worth noting:

- MangaDex titles are `Map<String, String>` localized objects; the plugin picks the first available value as primary and stuffs the rest plus `altTitles` into `aliases`.
- `MangaFormat` is inferred from `originalLanguage` plus the `Long Strip` / `Web Comic` / `Oneshot` / `Doujinshi` tag set, since MangaDex does not expose a single discriminator equivalent to AniList's `format`.
- `status` maps `ongoing‚Üíreleasing`, `completed‚Üífinished`, `hiatus‚Üíhiatus`, `cancelled‚Üícancelled`; anything else falls to `unknown`.
- Chapters with `chapter == null` (unnumbered prologues) are filtered out, since they cannot be ordered or addressed by `MangaChapterProgress.chapterNumber`. May revisit if a UX use case appears.
- Scanlator filter is post-applied because MangaDex's feed endpoint does not accept a scanlation-group filter directly.

HTTP envelope handling: `result == "ok"` is required; anything else maps to `mangadex.api_error`. 4xx/5xx map to `notFound` / `transport`. JSON decoding failures map to `mapping`. Search/list parse failures wrap in `mangadex.*_parse_failed` mapping errors.

**Verification:** `dart format` clean, `dart analyze` 0 issues at package and workspace level, 16/16 plugin tests pass. Tests use `package:http/testing.dart` `MockClient` against recorded JSON fixtures (no live network).

**Test infra detail worth remembering:** `http.Response(body, 200)` defaults to Latin-1 encoding when no `Content-Type` charset is set, which crashes the request-side validation when fixtures contain non-ASCII (e.g. `„ÉÅ„Çß„É≥„ÇΩ„Éº„Éû„É≥`). The test file wraps responses in a small `_ok(body)` helper that sets `content-type: application/json; charset=utf-8`. Future manga-source tests should follow the same pattern.

**Workspace:** registered the new package in the root `pubspec.yaml` workspace list.

**Residual risk:** the plugin is not yet wired to any repository ‚Äî it's verified only via fixture-backed unit tests. First real integration happens in Slice 8 (Manga Home/Search/Detail) where `MangaCatalogRepository` will compose AniList + matching + this plugin. The two areas most likely to need iteration are (a) MangaDex rate limiting (no per-request budget yet; will be added once the repository layer hits real load) and (b) the `_parseFormat` heuristic, which is best-effort given MangaDex's tag-based format model.

**Wave 1 ‚Äî `externalIds` on `SourceMangaMatch`** (commit `b329ccf`):

**Wave 2 ‚Äî `CompositeMangaCatalogRepository`** (commit `1dfc5cf`):
Lives in the app layer at `apps/kumoriya_app/lib/src/features/manga_catalog/application/services/`, parallel to `cached_anime_catalog_repository.dart`. `fetchMangaChapters(int anilistId)` is the meat. It resolves the AniList id to the source plugin's `sourceMangaId` in three steps:

1. **Strategy A**: search MangaDex by canonical romaji. Accept the row whose `externalIds['al']` equals the AniList id. This is the cleanest match and the only one that survives intentionally-mistitled releases.
2. **Strategy B**: fuzzy fallback ‚Äî case/whitespace-insensitive equality between any AniList title (romaji/english/native/synonyms) and the source row's title or aliases. Conservative; no `kumoriya_matching` dependency yet.
3. **Strategy C**: no match ‚Üí memoize `null`. Per Kumoriya rule #2 ("no source > wrong source"), an empty chapter list is preferred over guessing the wrong manga.

A `Map<int, String?>` in-memory cache memoizes both positive and negative resolutions for the session ‚Äî the `memoizes the resolved sourceMangaId` test asserts the second call to `fetchMangaChapters` does not re-search MangaDex. Persisting this mapping across launches (additive Drift column on `MangaCacheEntry.sourceMangaId`) is a deliberate follow-up: the in-memory map is enough to validate the pipeline and avoid the noisy migration in the same slice.

## 20:00 ‚Äî Player wakelock fix (Redmi Note 9 brightness auto-dim)

**Category:** fix
**Affected files:**
- `apps/kumoriya_app/pubspec.yaml`
- `apps/kumoriya_app/lib/src/features/player/presentation/pages/player_page.dart`

**Symptom:** A user on Redmi Note 9 reported that during playback the screen brightness drops on its own after a while, and tapping the screen brings it back up.

**Root cause:** Android's inactivity timer, not our code. MIUI is very aggressive ‚Äî at ~80% of the configured display timeout the OS dims the panel to a low level as a "screen is about to sleep" warning, and any touch resets the timer. While watching a video the user doesn't touch the screen, so the dim fires and the only way out is a tap. The bug is that the player never held a screen wakelock, so Android treated playback as idle.

**Fix:** Held a Flutter-managed screen wakelock for the player's lifetime. Added `wakelock_plus: ^1.2.10` (it was already a transitive dep at 1.5.1 from sentry/media_kit; pinned a direct floor that resolves to the same major). `_PlayerPageState.initState` calls `WakelockPlus.enable()`; `dispose` calls `WakelockPlus.disable()`. Both are `unawaited` ‚Äî failure to acquire the lock should never block the player from opening, and we don't expose any UI that depends on the lock state.

Scoped to `_PlayerPageState` (not the inner `_ImmersivePlayerView`) on purpose: the immersive view rebuilds on engine swaps, status transitions, and Windows-fullscreen toggles, but the player page is mounted exactly once per playback session, so the lock has the right lifetime there. No need to gate on `isPlaying` because the user is on the player UI even when paused ‚Äî keeping the screen on during a pause is the desired behavior, not a regression.

**Verification:** `flutter analyze` 0 issues; `dart format` clean. The fix can't be unit-tested because `WakelockPlus` is a platform-channel singleton without an injection seam in the app; manual smoke on Android (any device, but specifically the Redmi Note 9 reporter) is the validation step before the next release.

**Residual risk:**
- The wakelock is held during the player loading and error states too, not only while video plays. Acceptable ‚Äî those states are short-lived and we already keep the controls visible. If a future "player idle while waiting for ready" UX (e.g. host waiting in a watch party) holds the screen open for minutes, we may want to release the lock when `playbackEngine.state == idle` for >N s. Not in scope.
- On platforms where `wakelock_plus` is unsupported (none of our current targets, but worth noting) the call is a no-op; no try/catch needed.

**Out of scope (explicitly):**

- **Backend endpoints** for manga sync (Slice 10C-2). Until those exist, manga entries pile up locally ‚Äî fine for normal usage volumes (a few hundred kB per heavy user) but documented.

**Residual risk:**

- The local queue grows unbounded for manga until the backend lands. The Drift store has no eviction policy on `pending` and the queue is unbounded by design. A multi-month gap with very heavy usage could push the local DB into the few-MB range; not a hard problem but worth a poison cap if 10C-2 slips.
- `clearAllProgress` and `clearAllHistory` are pass-through on the manga side (mirrors anime behaviour during logout). Anime's wrapper enqueues per-row deletions for the still-authenticated history; the manga wrapper deliberately doesn't because the backend doesn't accept manga deletions yet ‚Äî when 10C-2 lands the manga wrapper needs to grow that path.
- The sync-aware decorators read the post-write `getEntrySnapshot` to build the payload. If a concurrent local write happens between the inner write and the snapshot read, the queue may carry the merged state ‚Äî which is fine semantically (LWW on the server) but means tests should keep their writes serialized.
