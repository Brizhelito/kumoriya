## 21:40 — Fix airing-notifications worker missing episodes across bucket boundaries

**Category:** fix
**Affected files:**
- `kumoriya-api/internal/notifications/airing_worker.go`
- `kumoriya-api/internal/notifications/airing_worker_test.go`

**Summary:**
The `AiringWorker` was calling `HomeService.AiringCalendar` with `Days: 1`, which translated to an upstream query with `airingAt_greater = now.Truncate(5min)`. AniList treats `_greater` as strict `>`, so episodes whose `airingAt` fell just before the current 5-min cache bucket were filtered out at the upstream before the worker's `[now-Window, now]` selector ran. Combined with the worker's 5-min `Tick`, recently-aired episodes systematically slipped through and, with `DedupTTL = 7d`, were never re-evaluated — matching the user-reported "no notifications for a week" symptom. Switched to an explicit window (`airingAtGreater = now.Truncate(5min) - 30min`, bucketed so the upstream cache key stays stable) and split the dispatch filter into `dispatchWindowStart`. Added `TestAiringWorker_RecoversAcrossBucketBoundary`; the legacy `fakeCalendar` was upgraded to honour `AiringAt{Greater,Lesser}` strict semantics so this class of bug can be reproduced in tests.

**Residual risk:** none on the worker itself. Pre-existing operational gap unaddressed: if FCM/Redis env vars are unset on the API, `startAiringWorker` silently logs a warn and returns nil — runtime monitoring of that warn is still on the user.

## 21:55 — Admin endpoint to send custom FCM pushes

**Category:** infra
**Affected files:**
- `kumoriya-api/cmd/server/main.go`
- `kumoriya-api/internal/handler/notifications_admin_handler.go`
- `kumoriya-api/internal/handler/notifications_admin_handler_test.go`

**Summary:**
Added `POST /internal/notifications/test` to dispatch a single FCM topic message with a custom `{topic, title, body, data}`. Gated by the existing `RELEASE_PUBLISH_TOKEN` bearer (constant-time compare, same `validBearerToken` helper used by release publishing) so no new shared secret is introduced. Extracted `buildFCMSender` from `startAiringWorker` so the worker and the admin endpoint share one Sender. Returns 503 if either the token or the sender is unconfigured, 401 on bad bearer, 400 on missing fields, 502 on FCM upstream errors. Useful as a smoke test to confirm topic delivery (e.g. `media_<anilistId>`) without waiting for an airing window.

**Residual risk:** the endpoint accepts arbitrary topic names, so anyone with `RELEASE_PUBLISH_TOKEN` can push to any topic clients subscribe to. Acceptable given the token is the deploy secret.

## 22:05 — Broadcast FCM push on release publish + client subscribes to `app_updates`

**Category:** feature
**Affected files:**
- `kumoriya-api/internal/notifications/topics.go`
- `kumoriya-api/internal/handler/release_handler.go`

- `apps/kumoriya_app/lib/src/shared/notifications/fcm_service.dart`
- `apps/kumoriya_app/test/fcm_topic_test.dart`

**Summary:**
After a successful `POST /internal/releases/publish`, `ReleaseHandler` now fans out a single FCM topic message to the new shared topic `app_updates` (constant in `notifications.AppUpdatesTopic`, mirrored as `kAppUpdatesTopic` in Dart). Title is `Nueva versión <version>`; body picks the first non-empty of `summary.es`, `summary.en`, `manifest_release_notes`, falling back to a default; data carries `{type:"app_update", version, tag, deep_link}`. Send is best-effort: an FCM error logs at warn and never rolls back the publish (manifest is the source of truth, push is just a hint). The `FcmService.initialize()` now subscribes every install to `app_updates` (idempotent; SDK queues until token is ready) so clients learn about new versions without polling the manifest. Tests assert exactly one broadcast per publish, the topic, the title, the Spanish-summary body preference, and the data payload.

**Residual risk:** clients without Google Play Services (e.g. de-Googled ROMs) still rely on the periodic manifest poll — by design.
