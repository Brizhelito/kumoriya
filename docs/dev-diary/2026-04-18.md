# 2026-04-18 — AniList Home edge-cache + FCM notifications backend

## Summary

Delivered the backend half of the AniList Home edge-cache and FCM airing
notifications pipeline. No Flutter-side changes yet; clients continue to
call AniList directly until Slice 2 lands.

## What shipped

### Slice 1 — AniList Home cache (Go)

Centralised three high-reuse Home surfaces (trending, season-discovery,
airing-calendar) behind an in-memory stale-while-revalidate cache.
Clients that opt in (Slice 2, pending) hit the API instead of
`graphql.anilist.co`, which keeps the tiny AniList rate limit budget
from being multiplied by thousands of users.

Per-media detail, search, lists and relations remain client-direct on
purpose: they are long-tail and benefit from distributed IPs, not from
central caching.

- `internal/anilist/` — queries mirror `packages/kumoriya_anilist`
- `internal/anilist/client/` — minimal GraphQL client
- `internal/anilist/cache/` — SWR cache with single-flight refresh and
  "fallback to stale on loader failure" behaviour so transient AniList
  outages stay invisible to users
- `internal/anilist/service/` — HomeService (Trending, SeasonDiscovery,
  AiringCalendar) with conservative TTLs (10 min / 30 min / 5 min fresh)
- `internal/anilist/scheduler/` — prewarm cron that runs immediately at
  boot and on cadence
- `internal/anilist/handler/` — three public REST endpoints under
  `/v1/anilist/home/*` with `Cache-Control` + `X-Cache-Stale`/`X-Cache-Age`
  observability headers

Tests: 5 unit tests for the SWR cache (cold miss, stale refresh, expired
fallback, cold-miss failure propagation, single-flight), 6 unit tests
for HomeService, and 4 integration tests against real AniList.

### Slice 5 — FCM airing notifications + Upstash dedup (Go)

Backend of the airing notifications pipeline. The Flutter side of this
slice (topic subscribe/unsubscribe on library toggle, notification
permission, deep-link handling) is not yet done.

- `internal/redis/` — minimal Upstash REST client (PING, SETNX, DEL).
  REST chosen over TCP because the load is trivial (a few commands per
  airing cycle) and it works identically from Cloudflare Workers if we
  ever want cross-runtime sharing.
- `internal/notifications/dedup.go` — RedisDeduper with keys
  `notif:sent:media:{id}:ep:{n}` and 7-day TTL. Keeps restarts of the HF
  Space from re-notifying users.
- `internal/notifications/fcm_sender.go` — wraps
  `firebase.google.com/go/v4` Messaging. Supports both env-var JSON
  (preferred for HF Spaces) and file-path credentials.
- `internal/notifications/airing_worker.go` — polls the cached calendar,
  filters to the "just aired" window, claims `(mediaID, episode)` in
  Redis, dispatches one FCM message per claim to `media_{anilist_id}`
  topics. Adult media is filtered. FCM failures on one media do not
  block dispatching the rest of the cycle.
- `cmd/server/main.go` — wires the worker behind a soft gate: if either
  Firebase or Upstash creds are missing, the worker stays disabled with
  a warning log so dev boots cleanly.

Tests: 6 unit tests (window filtering, dedup across 5 cycles, adult
skip, partial FCM failure tolerance, title fallbacks, empty payload,
service-account JSON validator) and integration tests against real
Upstash (PING + SETNX) and Firebase Admin SDK (sender constructs).

## Key decisions

- **AniList data split** — shared surfaces centralised, personalised
  surfaces stay client-direct. This keeps our server's AniList rate
  limit usage constant regardless of user count.
- **Memory-only Home cache** — Upstash was proposed for Home but the
  blast radius of a cache reset on HF restart is tiny (~4 refetches to
  warm up). Memory avoids a dependency we do not need.
- **Redis used only for notification dedup** — this is the one piece of
  state that, if lost on restart, causes a user-visible bug (duplicate
  pushes). Everything else is recoverable.
- **Subscriptions reuse existing `LibraryEntry.notify_new_episodes`** —
  no new tables, no new sync endpoints. Discovered the data model
  already supports this.
- **FCM topics, not user-token fan-out** — eliminates the need for the
  server to keep an authoritative `media -> users` index.

## Residual risk

- The airing worker has never sent a push to a real device yet.
  Slice 3+4 (Flutter FCM setup, topic subscribe on ♥) is required
  before end-to-end dispatch can be observed.
- `media_{id}` topic discipline assumes clients call
  `FirebaseMessaging.subscribeToTopic('media_{id}')` when
  `notify_new_episodes` flips to true. If Slice 4 forgets this, the
  server will dispatch into empty topics (no user impact, just no
  notifications).
- HomeService cache TTLs are first guesses. Real traffic will tell us
  if `trending` at 10-min fresh is too aggressive. Easy to tune.

## Later the same day — Slices 2, 4, 6 shipped

### Slice 4 — FCM topic subscriptions wired to library toggle

- `FcmAwareLibraryStore` decorator: every `setSubscription(id, notify:)`
  now mirrors to `FirebaseMessaging.subscribeToTopic('media_{id}')` /
  `unsubscribeFromTopic(...)`. Inner write failures short-circuit; FCM
  errors are swallowed so the library mutation always succeeds.
- `FcmTopicSyncService.syncTopicsWithLibrary()`: idempotent
  reconciliation invoked (a) after login migration+pull completes and
  (b) on every app boot from `AuthStateNotifier.build()` when a session
  is already active. Local→FCM only — FCM does not expose "list my
  topics", so stale subscriptions on their side are harmless and
  re-subscribing is a no-op.
- Tests: 6 for `fcm_topic.dart` (topic naming / parsing), 6 for
  `FcmAwareLibraryStore` (toggle, inner-write failure, FCM error
  swallowing, pass-through).

### Slice 2 — Backend-first AniList gateway + feature flag

- `AnilistHomeBackendClient`: public unauthenticated HTTP client for
  `/v1/anilist/home/{trending,season-discovery}` with a short
  per-request timeout. Maps network / HTTP / JSON errors into
  `AnilistError` values so the decorator can decide to fall back.
- `BackendFirstAnilistMetadataGateway`: overrides `fetchHomeCatalog`
  and `fetchSeasonDiscovery`; **everything else passes through
  unchanged** (including airing calendar — contract mismatch with the
  Go endpoint's `days` parameter vs Dart's `from`/`to` timestamps is
  a follow-up slice).
- Fallback behaviour: on any backend error (transport, non-2xx,
  malformed payload), transparently calls the inner direct-AniList
  gateway. Users never see an error caused by the Kumoriya backend
  being unavailable.
- Feature flag: `KUMORIYA_GO_ANILIST_HOME` (default true) via
  `--dart-define` for emergency rollback without rebuild of the
  backend.
- `KumoriyaRuntimeConfig.apiBaseUrl` exposes the Go backend URL so
  it can be overridden at build time via
  `KUMORIYA_API_BASE_URL`.
- Tests: 8 unit tests — success path, 502/malformed/network-error
  fallback for trending, three variants for season discovery, and a
  pass-through assertion for airing calendar (must never hit the
  backend).

### Slice 6 — Legacy worker trimmed, not deleted

**User flagged a real concern**: blindly removing the worker would
also kill auto-download of new episodes. The worker does two jobs —
local notifications (now redundant with FCM) AND auto-download
(still client-side, still needed).

Decision after discussion: **keep the worker** but reduce its
responsibilities.

- `check_new_episodes_worker.dart` rewritten: removed all
  `flutter_local_notifications` code paths, the
  `kCheckNewEpisodesDebugProbeTask` branch, and the
  `_runDebugProbe` / `_initNotifications` / `_sendNotification` /
  `_formatNotificationBody` helpers. 516 lines → 406 lines. Pure
  auto-download + warmup now.
- `main.dart`: `_initWorkmanager` cadence dropped from 1h to 4h.
  The cancel-on-FCM block was removed — the worker must keep
  running for auto-download. Comment updated.
- `settings_page.dart`: removed the debug "Test notificacion"
  button and `scheduleDebugBackgroundNotificationProbe` import.
- Channel-ID alignment bug fixed: server was sending
  `airing_episodes` while the Android client created
  `kumoriya_new_episodes`. Pushes would have landed in an
  auto-created channel with default importance instead of the
  configured one. Server now sends `kumoriya_new_episodes`.

**Why 4h and not FCM-driven retries?** Two reasons:
1. Sources publish episodes 30 min – 24 h after AniList airing time.
   An FCM-driven immediate attempt would almost always fail on fast
   sources. A state-machine of retries (30 min / 2 h / 6 h) was
   discussed but is too much complexity for this slice.
2. 4h worker cadence covers the entire source-publish window with
   ≤4h of extra latency past source availability. User's push
   notification is still instant; auto-download tolerates the gap.

Known limitation: N devices × 1 scrape per 4h still multiplies at
the source when many users track the same airing anime. Documented
as a future slice ("collaborative availability cache") in
`docs/architecture/notifications_and_auto_download.md`.
