# Kumoriya v0.3.0 — Alpha Release Notes

**Date:** 2026-04-24

## Summary

Release focused on downloads and distribution: a brand-new native download engine for Android, event-driven local/server sync, and a release pipeline that splits the APK per architecture so you only download what your device needs.

## Added

- **Android APK split per ABI** — we now publish separate APKs for `arm64-v8a`, `armeabi-v7a`, and `x86_64`, plus a `universal` fallback. The in-app updater auto-detects your architecture with `device_info_plus` and downloads the matching split (~35–45 MB) instead of the ~90 MB universal APK. The manifest (`/releases/latest`) and `update.json` now expose `universal` and a per-ABI `abis.*` map with `size_bytes` and `sha256` for integrity checks.
- **Native download engine rebuilt from scratch (Android)** — `kumoriya_exoplayer` ships a standalone OkHttp + coroutines download engine that replaces the legacy Media3 `DownloadManager` path. Direct MP4/WebM/MKV via a range-aware `ParallelDownloader`, HLS via segment fetch plus optional Media3 `Transformer` remux to MP4 (with `.ts` concat fallback on failure or low disk).
- **Download queue with concurrency cap** — at most three downloads run in parallel; the rest wait as `PENDING` in a FIFO queue and are promoted automatically as slots free up.
- **Auto-resume on cold start** — tasks that were `downloading`, `pending`, `remuxing`, or `disconnected` when the process died are re-enqueued on next launch and resume from `.state.json` + `.partial` bytes without re-downloading completed segments.
- **Downloads survive app close / swipe-away** — engine, notification center, and event sink are now process-scope; `FOREGROUND_SERVICE_DATA_SYNC` + `stopWithTask="false"` keep downloads running after the user swipes the app from recents.
- **Retry + error taxonomy** — `DownloadErrorClassifier` maps failures into `FailFast` / `RetryOnce` / `RetryBackoff` / `StorageFull` / `Disconnected`. 5xx and timeouts retry 1/2/4/8/16s (cap 5); 404 retries once; 401/403/410 fail fast. Terminal failures now carry a stable `errorCode` (`download.storage_full`, `download.server_error_503`, etc.).
- **WiFi-only + auto-resume on reconnect** — toggling "WiFi only" or losing connectivity auto-pauses active + queued tasks as `DISCONNECTED`; they resume automatically when an acceptable network returns.
- **Per-task + grouped download notifications** — each active download has its own notification with Pause/Resume/Cancel actions; a grouped summary exposes Pause all / Resume all.
- **Event-driven sync coordinator** — the new `SyncCoordinator` orchestrates six triggers: local writes with a 500 ms debounce, `AppLifecycleState.resumed` (full sync if the last sync is > 6 h old), network reconnection, `paused` scheduling a `OneTimeWorkRequest` on Android, blocking flush pre-logout (5 s timeout), and a periodic Workmanager task every 12 h as a fallback.
- **Pull independent of push** — new `SyncCoordinator.triggerPull()` calls only `pullSince(lastSyncAt)` under its own lock. Each foreground resume now pulls (30 s debounce), and a 30-min timer keeps pulling while the app is open so changes from other devices appear without waiting for the next close/open cycle.

## Changed

- **`media_kit_libs_video` replaced with desktop-only libs** — the Android APK no longer ships `libmpv / libavcodec / libdav1d` (~30–50 MB of `.so` saved). Android playback runs exclusively through `kumoriya_exoplayer` (Media3); `MediaKit.ensureInitialized()` is gated with `!Platform.isAndroid`.
- **`DownloadBackend` contract** — a single abstraction routes Android to `NativeDownloadBackend` and desktop to `DartDownloadBackend` (wraps the existing `DownloadManagerService`). Use cases and UI depend on the contract, not on concrete backends.
- **Unified HTTP stack** — player and downloader now share `KumoriyaHttpClient` (single OkHttp client with pool, redirects, timeouts, and UA extraction). Media3 playback uses `OkHttpDataSource.Factory` instead of `DefaultHttpDataSource`.

## Fixed

- **Downloads UI not updating** — the native flow now emits explicit `state` events (pending → downloading → completed/failed) and the Dart service reflects them immediately in Drift and Riverpod streams.
- **HLS downloads getting stuck** — Media3 was treating every stream as progressive and downloading only the `.m3u8`; HLS URLs now go through `HlsDownloadPreparer` with the correct MIME.
- **Early events lost** — the `EventChannel` sink is now mutable and buffered; events occurring before Flutter subscribes are drained on attach.
- **Cancel left downloads marked `failed`** — cancel now deletes the task from Drift and cleans up segments / temp cache (cancel = delete).
- **Duplicated custom headers in URL** when handing a download to the native plugin.
- **Queue stuck after cold start** — before the reconciliation pass, `pending`/`downloading` Drift rows had no matching native job; pause/resume were no-ops.
- **Swipe-away killed downloads** — Flutter detach used to destroy the engine and stop the FGS. Engine ownership moved to a process-scope singleton (`DownloadCore`).
- **`.ts` and `.mp4` artifacts orphaned on cancel** — the cancel path now sweeps both HLS remux variants plus their `.partial` siblings.
- **Auto-pause lost task on enqueue race** — the waiting queue is locked so concurrent enqueues cannot over-count active jobs; duplicate resume/retry clicks dedupe on `taskId`.

## Migration Notes

- None for end users. If your device's ABI is not one of the three supported, the updater falls back to the `universal` APK.
- Backend: this version adds the `app_release_android_artifacts` table with a backfill from the legacy universal column; backward-compatible with the previous shape.

## Platforms

- Android (per-ABI APK + universal)
- Windows (published separately if the installer for this version is not yet available)
